# Tauri Client UI 规范

## 1. 文档目的

本文档用于约束 `windows/tauri-client` 后续新增页面、功能和样式时的 UI 设计与实现方式。

目标不是限制开发效率，而是防止 UI 继续失焦，尤其防止以下问题再次发生：

- Element 只是名义上使用，实际大量绕开；
- 全局样式越写越宽，局部组件互相污染；
- 为了一个视觉效果重写整个控件；
- 同类交互在不同页面各写一套；
- 主题切换越来越依赖零散补丁。

---

## 2. 第一原则：Element Plus 是底层，不是摆设

后续做 UI 时，默认思路必须是：

> **先找 Element 是否已经提供结构、交互语义或状态能力；只有在默认视觉无法满足目标时，才做局部接管。**

这条原则适用于：

- 布局；
- 表单；
- 按钮；
- 滚动区；
- 弹层；
- 空态 / 错态 / 加载态；
- 选项组；
- 提示类交互。

### 2.1 禁止的做法

- 明明有 `el-button`，却先写原生 `button`；
- 明明有 `el-dialog`，却自写一套弹窗壳；
- 明明有 `el-scrollbar`，却把整个页面滚起来；
- 明明有 `el-empty` / `el-alert`，却随手写 `div` 提示。

---

## 3. 第二原则：先保证语义和行为，再追求视觉

控件的第一职责是正确表达语义和交互状态，其次才是像素效果。

例如：

- 主动作按钮应优先是 `el-button`；
- 禁用态应优先使用 Element 的 `disabled` 语义；
- 加载态应优先使用 Element 的 `loading` 语义；
- 选项切换应优先使用 `el-radio-group` / `el-tabs`；
- 表单输入应优先使用 `el-form` + `el-input` 体系。

也就是说，不允许为了“看起来像”而牺牲控件的可预期行为。

---

## 4. 第三原则：视觉收口顺序固定

后续新增 UI 时，视觉处理顺序必须固定：

### Step 1：先使用 Element 默认结构

先让页面或功能能跑，结构正确，交互正确。

### Step 2：通过 token 调整全局视觉语义

如果是颜色、背景、圆角、阴影、文本等级这类全局视觉问题，先回到：

- `windows/tauri-client/src/shared/styles/tokens.css:1`

不要直接在局部写死颜色。

### Step 3：通过 `element.css` 做有限桥接

只有当 Element 自身变量层面需要映射时，才改：

- `windows/tauri-client/src/shared/styles/element.css:1`

这个文件只允许做：

- Element 变量映射；
- 少量全局基础修正。

### Step 4：最后才做局部组件接管

如果仍无法满足目标视觉，再去对应局部样式文件中处理。

---

## 5. 第四原则：全局样式桥必须保持克制

`element.css` 不是“全站视觉垃圾桶”。

### 5.1 允许写入的内容

- `--el-*` 变量映射；
- Element 卡片 / 对话框 / popover 的统一基底色；
- 输入框 / wrapper 的通用背景与 focus 边界；
- 少量文本、分割线、tag 之类的基础面修正。

### 5.2 不允许写入的内容

- 全站所有按钮的强视觉接管；
- 某个具体页面才需要的布局；
- 某个单独功能特有的视觉规则；
- 依赖某个业务状态的颜色分支；
- 临时修某个页面问题的补丁。

如果一条样式只服务于某一页或某一功能，它就不该进 `element.css`。

---

## 6. 第五原则：局部接管必须带边界

局部接管不是不能做，但必须做到“范围可控、意图清楚”。

### 6.1 正确方式

用明确的局部 class 包裹后再覆写，例如：

- `app-dialog-tabs`
- `app-dialog-chips`
- `settings-chip-group`
- `disk-card__primary-action`

### 6.2 错误方式

- 直接覆写 `body .el-tabs__item`
- 直接覆写 `body .el-radio-button__inner`
- 直接把全局 `el-button` 改成项目视觉

### 6.3 接管时必须回答的问题

在决定接管某个 Element 控件前，先问三件事：

1. 这个效果能不能用 token 或变量桥解决？
2. 这个视觉差异是局部需求，还是全局需求？
3. 这次接管是否依赖 Element 内部 DOM 细节？

如果第 3 条答案是“是”，必须再想一遍是否有更稳妥的方式。

### 6.4 如果必须接管内部包装层，要接管“真实生效层”

像 `el-button` 这类控件，Element 可能会给插槽内容增加内部包装层。

这时要注意：

- 按钮外层的 `flex-direction` 不一定能真正控制内容排布；
- 需要先确认当前版本下真正承载插槽内容的是哪一层；
- 然后只在局部 class 范围内接管那一层。

当前已经踩实过的一个具体例子是：

- 顶部会话状态按钮里，“绿圆 / loading 图标 + 文本”的间距，单改按钮外层 `gap` 可能没有效果；
- 真正生效的往往是按钮内部实际承载插槽内容的 `span`；
- 因此这类场景应优先检查局部按钮的真实直接子内容层，必要时同时兼容 `.el-button__text`，而不是只改外层后误判“CSS 失效”。

正确目标是：

- **接管真实生效层**

而不是：

- 猜一个看起来像内部实现的类名再去赌它会一直有效

---

## 7. 第六原则：避免依赖 Element 内部结构补偿

后续样式实现时，尽量不要写这类规则：

- `nth-child(...)`
- 依赖某个内部包裹层顺序的补丁；
- 靠层层嵌套选择器压住 Element 默认样式；
- 靠大量 `!important` 进行结构性修复。

原因很直接：

- Element 升级后容易失效；
- 很难解释为什么这样写；
- 新功能复制时容易把脆弱实现扩散出去。

可以接受的例外是：

- Element 默认结构没有开放更稳定的挂点；
- 需求当前必须实现；
- 并且接管范围已经被局部 class 明确包住。

即便如此，也应优先选择“稳定 class + 局部变量覆盖”的方式。

### 7.1 关于 `!important` 的明确约束

后续允许使用 `!important` 的前提必须同时满足：

1. Element 默认优先级或内联结构确实压不住；
2. 目标视觉已经明确，不是临时试错；
3. 规则范围已经被局部 class 包裹；
4. 没有更干净的 token / 变量 / 结构方案。

以下场景通常不允许直接上 `!important`：

- 为了偷懒覆盖全局按钮；
- 为了修一个页面偶发样式错位；
- 为了在多个文件之间抢优先级；
- 为了规避对结构问题的正面修复。

一句话：**`!important` 可以存在，但只能是局部接管工具，不能变成常规实现手段。**

---

## 8. 第七原则：颜色与尺寸不允许散落硬编码

### 8.1 颜色

颜色优先来源于 token，而不是直接写字面值。

可直接使用的通常应是：

- `--bg-*`
- `--border-*`
- `--text-*`
- `--accent*`
- `--success`
- `--warning`
- `--danger`

### 8.2 尺寸

圆角、阴影、过渡时长也应优先复用 token：

- `--radius-*`
- `--shadow-*`
- `--transition-*`

### 8.3 允许例外

以下场景可以少量硬编码：

- 特定图标色；
- 某个状态按钮明确要求独立语义色；
- 当前 token 体系确实还未覆盖该语义。

但如果同一颜色在多个地方重复出现，就说明它应该被抽成 token。

### 8.4 主题差异优先回到 token 层

如果后续出现以下需求：

- 某个亮色主题字体需要更暗；
- 某个暗色主题强调色需要更亮；
- loading 色在某一色家族下不够清楚；

优先处理位置应是：

- `tokens.css` 的主题变量；
- 必要时新增更细的语义 token。

不推荐的做法是：

- 在具体 feature 组件里写主题分支；
- 针对某个按钮单独写 `body[data-theme-color=...]` 的局部补丁；
- 在多个样式文件里分别修同一主题问题。

---

## 9. 第八原则：布局问题必须在布局层解决

不要把布局问题下沉到业务组件里乱修。

### 9.1 正确做法

- 页面骨架问题改 `layout.css`
- 顶部栏问题改 `header.css`
- 列表滚动问题改 `listPanel.css`
- 对话框结构问题改 `dialogs.css`

### 9.2 错误做法

- 滚动条挡住卡片，就给卡片单独乱加 margin；
- 页面太窄，就在子组件里硬写 width 补丁；
- 列表高度问题，靠页面整体滚动糊过去。

布局问题必须回到容器层和滚动层收。

### 9.3 页面骨架层的强制覆盖要集中管理

如果必须对 Element 容器的默认 `padding` / `height` 做强制覆盖，应尽量集中在：

- `layout.css`

不要在：

- 各 page 组件各写一套；
- 各 widget 再重复压一遍；
- 某个 feature 为了显示正常自己改外层容器。

页面壳问题必须在页面壳层解决。

---

## 10. 第九原则：业务状态通过组件语义表达，不靠纯样式猜

例如：

- 正在挂载 / 卸载中：优先使用按钮 `loading`
- 不可操作：优先使用 `disabled`
- 无效盘：优先通过显式状态驱动 UI 分支
- 提示信息：优先使用 `el-tooltip` / `el-alert` / `ElMessage`

禁止出现：

- UI 纯靠颜色猜测状态；
- 某个 `class` 既表达业务状态又承担布局职责；
- 同一状态在不同地方靠不同字段判断。

UI 层必须消费明确状态，而不是自行推断。

### 10.1 运行时介质持有也必须保持单一真状态

后续涉及磁盘挂载、卸载、重扫、无效恢复时，必须遵守：

- 每个 `DiskRuntime` 只保留一份唯一 `media` 字段；
- 不允许再把“内存盘持有介质”和“文件盘持有介质”拆成两套并行字段；
- UI 不额外维护“这盘当前是否持有 media”的镜像状态。

当前固定语义：

- 内存盘：`media` 常驻，卸载后回收到同一字段；
- 文件盘：卸载或无效时 `media` 置空，重连时按路径即时重建；
- 无效盘：没有 `media` 实例。

如果后续要做新功能，例如：

- 重扫；
- 自动挂载；
- 文件类型扩展；
- 网络盘；

都必须在这个单一持有模型上继续扩展，而不是重新引入第二份介质状态。

### 10.2 `loading` 颜色跟随当前动作语义，不回退全局主题色

如果一个按钮本身已经有明确业务语义色，例如：

- 挂载 = 绿色
- 卸载 = 红色
- 危险删除 = 危险色

那么它进入 `loading` 时，应采用：

- **当前动作颜色的灰化延续**

而不是：

- 统一退回主题主色的 loading 表现

原因是统一退回主题色会制造语义错位：按钮明明表示“挂载中”或“卸载中”，结果却突然变成与当前动作无关的主题色系。

因此后续规则固定为：

- 普通主题主按钮：`loading` 可跟随主题主色
- 业务语义按钮：`loading` 必须跟随该按钮自己的语义色

### 10.3 启动覆盖必须停留在主页单一映射口

如果后续继续扩展主页启动链、重扫链或会话失败展示，必须遵守：

- `正在初始化 / 会话失败` 只属于主页展示覆盖；
- Backend 真实磁盘状态不新增对应状态值；
- 磁盘卡片组件不自行判断是否覆盖；
- API DTO 不为这层覆盖再补一套平行状态字段。

唯一允许的做法是：

- 宿主继续提供真实 `runtimeDisks`；
- 主页单一映射口生成 `displayDisks`；
- 组件只消费最终展示结果。

### 10.4 会话状态入口与重试链必须保持单一职责

后续如果继续扩展会话相关 UI，必须遵守：

- 顶部会话状态按钮是查看当前会话阶段的唯一正式入口；
- 会话状态对话框是失败后执行重试的唯一正式入口；
- 重试链只重试 `open_session -> rescan_runtime_disks -> 自动挂载`；
- 重试不重复执行 `restore_client_state`；
- 托盘重新显示主窗口不视为一次新启动，不重复触发首次自动挂载。

## 11. 第十原则：新功能 UI 的落地流程

后续加任何一个新功能，建议按下面流程推进：

### 11.1 先画出最小结构

先确定：

- 这是 page、widget 还是 feature；
- 需要哪些 Element 基础组件；
- 哪些区域是主动作，哪些是次动作；
- 哪些状态是 loading / disabled / invalid / empty / error。

### 11.2 先做无样式闭环

先让：

- 页面能显示；
- 功能能走通；
- 状态能切换；
- 空态、错态、加载态都有。

### 11.3 再做视觉贴合

这个阶段再判断：

- token 是否够用；
- 需不需要扩展局部样式；
- 需不需要接管某个 Element 控件。

### 11.4 最后做统一性检查

检查四件事：

1. 有没有绕开 Element 去自造基础控件；
2. 有没有把局部需求写进全局样式桥；
3. 有没有新增脆弱结构补偿；
4. 有没有把颜色 / 尺寸写散。

### 11.5 再加一项复查：有没有制造可预见的重复样式

如果新功能里出现以下情况，需要停一下再判断：

- 又出现一套新的 chip 单选风格；
- 又出现一套新的 tabs 视觉；
- 又出现一套新的卡片动作按钮规则；
- 又出现一套新的弹窗 header / footer 结构。

如果只是同类模式的轻微变体，应优先考虑：

- 复用已有模式；
- 抽共享视觉块；
- 调整 token 或局部 modifier。

不要让“类似但不完全一样”的重复实现继续增殖。

---

## 12. 推荐的文件落点

后续 UI 相关修改，优先按下面方式落：

### 12.1 主题 / 全局语义

- `windows/tauri-client/src/shared/styles/tokens.css:1`

### 12.2 Element 变量桥 / 基础面修正

- `windows/tauri-client/src/shared/styles/element.css:1`

### 12.3 页面与功能样式

- `windows/tauri-client/src/shared/styles/layout.css:1`
- `windows/tauri-client/src/shared/styles/header.css:1`
- `windows/tauri-client/src/shared/styles/listPanel.css:1`
- `windows/tauri-client/src/shared/styles/diskCard.css:1`
- `windows/tauri-client/src/shared/styles/dialogs.css:1`
- `windows/tauri-client/src/shared/styles/settingsPage.css:1`

### 12.4 组件与功能代码

- 页面结构：`src/pages`
- 大块 UI：`src/widgets`
- 用户动作闭环：`src/features`

不要把“视觉逻辑”和“业务流程逻辑”混在同一个巨文件里。

---

## 13. 允许的设计自由度

本规范不是要求所有新 UI 都长得一样，而是要求新增 UI 的实现方式一致。

允许变化的内容：

- 区块布局；
- 视觉层次；
- 业务动作分布；
- 局部特殊态；
- 主题配色表现。

不允许变化的内容：

- 底层控件体系到处切换；
- 同类问题每次都换一种实现；
- 一遇到视觉差异就直接重写控件。

---

## 14. 简明执行版

后续改 UI 时，按这个最短版本执行即可：

1. **先用 Element 组件搭起来**
2. **先走通状态和交互**
3. **颜色尺寸先看 token**
4. **全局桥只做变量映射和基础面**
5. **局部视觉只在局部 class 下接管**
6. **不要绕开 Element 自造基础按钮/输入/弹窗**
7. **不要依赖 Element 内部结构补丁**
8. **布局问题回布局层解决**
9. **`!important` 只在局部受控接管时使用**
10. **相同视觉模式出现第三次时，考虑抽共享模式**

如果这 8 条都满足，后续新增功能的 UI 一般不会跑偏。
