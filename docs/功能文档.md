# 功能文档

## 1. 功能范围

本系统提供一个由应用层控制的虚拟磁盘能力集合，包含盘管理、事件分发、读写回包和异常退出删盘四类能力。

## 2. 组件功能

## 2.1 MyCtl 功能

`MyCtl` 基于当前 `MyKWDF3` 改造，承担控制平面职责。

### 必选功能

- 暴露自定义设备接口 GUID，供 `MyApp` 枚举
- 仅允许一个 App 会话独占打开
- 感知 `CreateFile`、`Cleanup`、`Close`
- 内核中定位 `MyDisk` 对应的 storageport 接口
- 代理 `APP_CMD` 和 `WAIT_EVENT`
- 在 App 退出时触发 `REMOVE_ALL_DISKS`
- 维护会话状态和心跳状态

### 不做的事情

- 不处理普通文件读写
- 不拦截文件系统栈
- 不实现虚拟盘数据面

## 2.1.1 启动自检配合

`MyCtl` 必须配合 App 的启动自检策略，满足以下要求：

- 系统中只能有一个有效 `MyCtl` 设备实例
- 仅允许一个 App 会话打开
- 对重复控制实例保持可识别
- 支持 App 通过查询命令识别“这是不是目标 MyCtl”

## 2.2 MyDisk 功能

`MyDisk` 基于当前 `MyDisk2` 改造，承担真实虚拟盘与 SCSI I/O 职责。

### 必选功能

- 暴露多个 Target，每个 Target 只有 LUN 0
- 默认所有 Target 均为 not present
- 支持建盘和删盘
- 支持 `INQUIRY`
- 支持 `READ_CAPACITY`
- 支持 `READ_CAPACITY16`
- 支持 `READ/WRITE/READ16/WRITE16`
- 支持 `REPORT_LUNS`
- 支持私有控制协议入口
- 支持事务表、事件队列和等待队列
- 支持总线变化通知

### 运行约束

- 无 App 会话时不暴露任何盘
- 只允许数据盘场景
- 任何盘都不能被用于系统盘、启动盘、分页盘

## 2.3 MyApp 功能

`MyApp` 基于当前 `MyApp5` 改造，承担用户态内存后端和控制客户端职责。

### 必选功能

- 单实例运行
- 启动时扫描驱动与设备状态
- 自动安装缺失驱动包
- 自动创建缺失设备实例
- 自动清理多余设备实例
- 枚举 `MyCtl` 的设备接口 GUID
- 建立独占控制会话
- 周期性发送心跳
- 维持 `WAIT_EVENT` 预投队列，深度默认为 64
- 请求创建磁盘
- 请求删除磁盘
- 维护用户态内存映像
- 处理系统读请求并回送数据
- 处理系统写请求并落到用户态内存映像
- 支持 App 主动退出前删盘

## 3. 功能清单

## 3.1 盘管理

### 创建磁盘

输入：

- 目标 `TargetId`
- 容量
- 扇区大小
- 可选标志

输出：

- 成功或失败
- 失败原因

约束：

- 必须已有控制会话
- 必须已有足够的 `WAIT_EVENT` 深度
- 一个 `TargetId` 同时只能对应一块盘

### 删除磁盘

输入：

- 指定 `TargetId`，或删除全部

行为：

- 立即将目标盘标记为 not present
- 取消并失败完成该盘所有未决事务
- 通知系统总线变化

说明：

删盘行为按“物理拔掉U盘”的激进策略处理，不保证文件系统状态收敛。

## 3.1.1 驱动与设备自修复

App 每次启动都必须做如下动作：

1. 检查 `MyCtl` 驱动包是否已安装
2. 检查 `MyDisk` 驱动包是否已安装
3. 如缺失，则安装驱动包
4. 检查 `MyCtl` 设备实例数量
5. 检查 `MyDisk` 设备实例数量
6. 如数量为 0，则创建设备实例
7. 如数量大于 1，则保留一个规范实例，删除多余实例

规范实例选择规则建议如下：

- 优先保留已启用且无 problem code 的实例
- 若存在多个健康实例，保留设备实例 ID 最小的一个
- 其余实例全部删除

## 3.2 会话管理

### 会话建立

- `MyApp` 打开 `MyCtl`
- `MyCtl` 建立唯一会话
- `MyCtl` 初始化对 `MyDisk` 的代理通道

### 会话保活

- `MyApp` 周期性发送心跳
- `MyCtl` 检查心跳超时
- `MyCtl` 监控 `WAIT_EVENT` 深度是否跌空

### 会话销毁

以下任一情况都视为会话结束：

- App 主动关闭句柄
- App 进程崩溃
- 心跳超时
- `WAIT_EVENT` 长时间枯竭

结果：

- 立即删掉全部磁盘

## 3.2.1 App唯一性

App 每次启动时必须先争抢单实例锁。若发现已有活动进程，则：

- 不允许再启动第二个工作进程
- 可选择激活已存在实例，或直接退出

本项目按更保守策略处理：

- 发现已有实例后直接退出

## 3.3 系统写盘流程

1. 系统向 `MyDisk` 发出写请求
2. `MyDisk` 生成 `WRITE_REQ`
3. `MyDisk` 为该请求分配事务 ID
4. `MyDisk` 通过 `WAIT_EVENT` 将事件发送给 `MyApp`
5. `MyApp` 将数据写入自己的内存后端
6. `MyApp` 通过 `APP_CMD_WRITE_ACK` 回包
7. `MyDisk` 完成原始写请求

## 3.4 系统读盘流程

1. 系统向 `MyDisk` 发出读请求
2. `MyDisk` 生成 `READ_REQ`
3. `MyDisk` 为该请求分配事务 ID
4. `MyDisk` 通过 `WAIT_EVENT` 将事件发送给 `MyApp`
5. `MyApp` 从自己的内存后端取出数据
6. `MyApp` 通过 `APP_CMD_READ_REPLY` 回包
7. `MyDisk` 把数据复制给系统并完成原始读请求

## 3.5 事件通道

`WAIT_EVENT` 是事件拉取通道，既承载读请求事件，也承载写请求事件。

事件头必须包含：

- 事件类型
- 事务 ID
- 目标 `TargetId`
- 起始 `LBA`
- 扇区数
- 数据长度
- 标志位

其中：

- 读事件不携带数据正文
- 写事件可携带内联数据，或只携带头部后由 App 二次拉取

## 4. 非功能要求

## 4.1 稳定性

- App 不在线时绝不保留盘
- App 启动时必须把驱动与设备状态收敛到唯一有效形态
- 驱动必须能处理 App 异常退出
- 事务超时必须可恢复

## 4.2 可调试性

- 三个组件都要带清晰日志
- 每个事务都可打印 `txId`
- 每个盘都要有 `TargetId`

## 4.3 并发性

- 支持多盘
- 支持并发系统I/O
- 支持并发 `WAIT_EVENT`
- 只允许单 App 控制会话

## 4.4 唯一性

- `MyApp` 进程唯一
- `MyCtl` 设备实例唯一
- `MyDisk` 设备实例唯一
- `MyCtl` 会话唯一
