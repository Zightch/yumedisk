# 项目概述

## 1. 项目目标

YumeDisk 要实现一个由用户态 App 主导的数据盘型内存虚拟磁盘系统。Windows 看到的是 `YumeDiskSCSI` 暴露的 Storport SCSI target；真实存储介质、读写一致性策略和磁盘生命周期由用户态 App 负责。

最终收口系统由三层组成：

- `RWTestApp`：用户态介质所有者、驱动与设备自检入口、建盘删盘控制端、per-disk slot engine。
- `YumeDiskKMDF`：控制驱动，负责自定义设备接口、单会话、watchdog、file cleanup 和 App direct I/O buffer 到 miniport 的 transport。
- `YumeDiskSCSI`：Storport miniport，负责 SCSI 枚举、系统 READ/WRITE SRB、per-target 队列和原始 SRB completion。

早期设计文档里的 `MyApp`、`MyCtl`、`MyDisk` 是历史逻辑名；正式文档统一使用 `RWTestApp`、`YumeDiskKMDF`、`YumeDiskSCSI`。

最终目标不是传统内核自包含 ramdisk，而是 App-owned media 架构：

```text
Windows storage stack
  -> YumeDiskSCSI per-target READ/WRITE queue
  -> YumeDiskKMDF session/direct-I/O slot transport
  -> RWTestApp per-disk medium and slot engine
```

## 2. 使用范围

该系统只面向以下场景：

- 普通数据盘。
- 应用软件直接读写的数据卷。
- 测试型、专用型、受控环境中的虚拟存储。

明确不支持以下场景：

- 系统盘。
- 启动盘。
- 分页盘。
- 依赖用户态回包才能维持系统稳定的核心卷。

## 3. 设计原则

### 3.1 App 不在线即无盘

只要没有有效 App 会话，`YumeDiskSCSI` 就不应暴露任何数据盘。App 正常退出、异常退出、heartbeat 超时或 KMDF file cleanup 都必须收敛到同一结果：清盘、取消 pending slot、失败完成 pending SRB。

### 3.2 启动即自检与自修复

App 启动后不能假设系统已经处于正确状态，而是先执行环境收敛：

- 检查 `YumeDiskKMDF` 和 `YumeDiskSCSI` 驱动包。
- 检查两个 Root/devnode 设备实例是否存在且唯一。
- 缺失时安装驱动包并创建设备实例。
- 重复时删除多余实例。
- 最后只打开 `YumeDiskKMDF` 自定义设备接口。

### 3.3 控制入口和 SCSI 入口分离

App 不直接枚举或打开 `GUID_DEVINTERFACE_STORAGEPORT`。`YumeDiskKMDF` 对外提供稳定控制入口，并在内核中通过 `QUERY_INFO` 等私有命令确认目标 `YumeDiskSCSI` miniport。

### 3.4 数据面只有一条链路

最终数据面只允许 app-owned slot queue：

- `POST_READ_SLOT`
- `POST_WRITE_SLOT`
- `READ_ACK`
- `WRITE_ACK_BATCH`

旧 `WAIT_EVENT -> inline data -> READ_REPLY/WRITE_ACK` 模型只作为历史基线和故障排查背景，不再作为目标实现。

### 3.5 状态归属唯一

- `YumeDiskKMDF` 只拥有 session、watchdog、miniport handle 和 App request 生命周期。
- `YumeDiskSCSI` 最终只拥有 per-target posted slots、pending reads/writes、eventId 和 SRB completion。
- App 最终只拥有 per-disk medium、slot engine、queue depth、READ_ACK 和 WRITE_ACK_BATCH flush。

不要回到 adapter 级共享数据面队列，也不要让多盘共享 App 全局 worker pool。

### 3.6 驱动不做 LBA overlap 一致性

读写在驱动层完全并发处理。驱动不做 LBA overlap 检测、不做读写排序；重叠区间的一致性由 App 自己负责。

## 4. 运行流程

### 4.1 启动阶段

1. `RWTestApp` 抢占单实例锁。
2. `RWTestApp` 收敛驱动包和 Root/devnode 实例。
3. `RWTestApp` 枚举并打开 `YumeDiskKMDF` 自定义设备接口。
4. `YumeDiskKMDF` 创建唯一 active session。
5. `YumeDiskKMDF` 打开同会话生命周期的 `YumeDiskSCSI` miniport handle，并通过 `QUERY_INFO` 确认目标。
6. `RWTestApp` 开始 heartbeat。

### 4.2 创建磁盘

1. App 通过控制命令请求创建目标 `TargetId`。
2. `YumeDiskKMDF` 将短控制命令代理给 `YumeDiskSCSI`。
3. `YumeDiskSCSI` 标记 target present 并通知总线变化。
4. 系统枚举出对应普通数据盘。
5. App 为该盘启动独立 slot engine，持续预投 read/write slots。

### 4.3 正常读路径

1. Windows 向 `YumeDiskSCSI` 发起 READ SRB。
2. `YumeDiskSCSI` 分配 `eventId`，把请求挂入该 target 的 `PendingReadsByEventId`。
3. `YumeDiskSCSI` 消费一个 posted read slot，填写 `eventId/lba/count`，经 `YumeDiskKMDF` 完成给 App。
4. App 从自己的 medium 读取数据。
5. App 通过 `READ_ACK` 把数据回送给驱动。
6. `YumeDiskSCSI` 将数据复制到原始 SRB buffer 并完成 READ SRB。

### 4.4 正常写路径

1. Windows 向 `YumeDiskSCSI` 发起 WRITE SRB。
2. `YumeDiskSCSI` 分配 `eventId`，按 write slot 容量拆成 `seq/totalSeq` fragment。
3. `YumeDiskSCSI` 消费 posted write slots，把 fragment 数据填入 App slot。
4. App 将 fragment 写入自己的 medium。
5. App 用独立 `WRITE_ACK_BATCH` 按 `eventId + seq range` 落账。
6. `YumeDiskSCSI` 用 bitmap 记录 ACK；全部 fragment 落账后完成原始 WRITE SRB。

### 4.5 退出与清理

1. App 正常关闭、异常退出、heartbeat 超时或 KMDF file cleanup 触发 session close。
2. `YumeDiskKMDF` 向 `YumeDiskSCSI` 发送 `RemoveAllDisks | SessionClose`。
3. `YumeDiskSCSI` 清盘，取消 pending slots，失败完成 pending SRB。
4. `YumeDiskKMDF` drain in-flight slot transport，关闭 miniport handle，并锁定该 file object 后续 IOCTL。

## 5. 交付结果

项目完成后应具备以下结果：

- App 启动后能自动把驱动与设备状态收敛到唯一正确形态。
- 应用层可动态创建和删除多个虚拟盘。
- 系统可将这些虚拟盘识别为普通数据盘。
- 读写路径由 App-owned slot queue 主导。
- App 退出后虚拟盘立即消失，pending I/O 被明确失败完成或取消。
- 多盘按多 Target 方式暴露，不使用多 LUN 方案。
- `Q1T1 / Q8 / Q32` 压测不再出现 100% 无读写假死、盘被误判死亡、会话残留或多盘全局串扰。
