# 传输层协议

## 1. 分层原则

所有连接统一走 TCP。

协议分为两层：

- 传输层：只负责 TCP 拆帧
- 业务层：完整封装在传输层数据区中，负责认证、路由、读写、偏移、长度等语义

传输层不承载任何业务语义。

## 2. 帧格式

传输层帧定义：

```text
frame = u16be payload_size_m1 + payload[payload_size_m1 + 1]
```

说明：

- `payload_size_m1` 为 2 字节无符号整数，按大端编码
- payload 实际长度为 `payload_size_m1 + 1`
- 取值 `0x0000` 表示 payload 实际长度为 `1`
- 取值 `0xFFFF` 表示 payload 实际长度为 `65536`

因此传输层数据区长度恒为：

```text
1..65536 bytes
```

不存在空包。

## 3. 传输层职责边界

传输层只做三件事：

1. 从 TCP 字节流中读取 2 字节长度头
2. 计算本帧 payload 实际长度
3. 继续读取完整 payload，并把整帧交给业务层

传输层明确不做：

- 鉴权
- 盘路由
- 压缩
- 加密
- 校验和
- 多帧重组
- 业务 continuation

## 4. 解析口径

单帧解析步骤：

1. 读取 2 字节 `payload_size_m1`
2. 计算 `payload_size = payload_size_m1 + 1`
3. 再读取 `payload_size` 字节
4. 将完整 payload 交给业务层

如果连接在长度头或 payload 未收完整前断开，则视为连接异常并直接终止当前连接。

## 5. 包粒度与并发

YumeDisk 当前目标是单端口打通，因此传输层必须限制单帧颗粒，避免超大包长期占用连接与缓冲区。

当前上限：

- 单帧业务数据最大 `65536` 字节

这个上限的意义：

- 颗粒足够小，便于提高单端口并发能力
- 缓冲区上限明确，便于两端做固定 buffer 复用
- 不需要引入更复杂的传输层分片协议

## 6. 业务层承载规则

一条业务消息必须完整落在一帧 payload 中。

这意味着：

- 一个业务请求对应一帧
- 一个业务响应对应一帧
- 传输层不允许把一个业务消息拆成多帧后再由对端重组

后续如果业务层定义固定头、操作码、偏移、长度、数据区，它们都属于 payload 内部内容，不属于传输层。

## 7. 第一版业务头约束

为支持多 `DiskSession` 并发复用同一条 `client -> gateway` TCP 连接，第一版业务层必须带固定请求头。

第一版约束字段：

```text
message = header + body

header {
  op_code
  request_id
  session_id
}
```

字段职责：

- `op_code`：标识当前业务命令类型
- `request_id`：用于 client 与 server 配对请求和响应
- `session_id`：标识目标远端盘会话

约束：

- `request_id` 由请求发起方生成
- 响应必须带回原始 `request_id`
- 除认证前命令外，后续业务命令都必须带 `session_id`
- 同一条 TCP 连接上允许多个 `request_id` 并发飞行
- 响应允许乱序返回
- 配对责任属于业务层，不属于传输层

认证前命令例外：

- `AuthStart`
- `AuthFinish`
- `SessionOpen`

其中：

- `AuthStart` / `AuthFinish` 不带 `session_id`
- `SessionOpen` 在打开会话前也不带 `session_id`

## 8. 大块 I/O 拆分规则

对大块盘操作，双方业务层必须在编码前主动拆分。

规则：

1. 业务层先计算当前消息编码后的总长度
2. 若总长度超过 `65536`，则必须拆成多个完整业务请求
3. 每个拆分后的请求都必须自带完整语义，例如自己的操作类型、偏移、长度和数据段
4. 对端只按单帧处理，不承担跨帧重组责任

这条规则同时适用于：

- `READ`
- `WRITE`
- 未来任何可能携带大块数据的业务操作

## 9. v1 工程建议

第一版实现建议：

- 每个连接至少准备可容纳 `65538` 字节的收包缓冲区
  - `2` 字节长度头
  - `65536` 字节 payload
- 通过 buffer 池复用收发缓冲区
- 由业务层再根据自身头部和字段定义，进一步限制 `max_io_bytes`

也就是说：

- `65536` 是传输层硬上限
- 业务层第一版应通过 `SessionOpenResponse.max_io_bytes` 下发更小的单次 I/O 上限
- 但业务层不能突破这个传输层上限
