# 单盘吞吐恢复方案草案

## 背景

以 `430e364` 为对比基准，近期多盘并发改造后出现了新的性能现象：

- 单盘顺序读写吞吐从约 `500MB/s` 降到约 `250MB/s`
- 多盘并发时，每个盘仍大致维持 `250MB/s`
- 多盘总吞吐明显超过 `500MB/s`

这个现象说明当前瓶颈不是全局共享瓶颈，而是 **每个盘各自存在一个独立瓶颈**。

## 当前判断

当前最可疑的瓶颈位于 App 侧每盘数据面的执行模型。

旧模型（`430e364`）大致是：

- 每盘 `Q` 个读 worker
- 每盘 `Q` 个写 worker
- 每盘 1 个独立写 ACK flush worker

新模型则收敛成：

- 每盘 1 个 `slot engine` 线程
- 由这个线程统一负责：
  - 投递 `POST_READ_SLOT`
  - 投递 `POST_WRITE_SLOT`
  - 等待完成
  - 从 medium 读/写数据
  - 发送 `READ_ACK`
  - 发送 `WRITE_ACK_BATCH`
  - 补投下一批 slot

这会把每盘数据面从“多执行体并发消费队列”退化成“多个挂起 slot + 单线程串行消费完成”。

换句话说，虽然表面上队列深度还是 `Q=32`，但真正跑热路径的执行体只剩 1 个，所以单盘容易卡在一个固定吞吐上限附近。

## 结论

当前单盘从 `500MB/s` 掉到 `250MB/s`，主因大概率是：

- App 每盘数据面并发度下降
- `READ_ACK` / `WRITE_ACK_BATCH` 与 slot 完成处理串在同一个线程里
- 当前循环每次只消费一个完成事件，完成事件 drain 不够激进

当前不优先怀疑的方向：

- KMDF 默认控制队列全局串行
- SCSI 全局队列深度共享
- 全局总带宽上限

因为如果这些是主因，多盘总吞吐不应该明显超过 `500MB/s`。

## 重建目标

不回退到旧版整套实现，但恢复每盘真实并发能力。

目标是重建为“每盘双平面”：

- 读平面
- 写平面

每个盘至少由两个长期运行的数据面线程负责，而不是一个线程包办整条路径。

## 目标结构

### 1. 读平面

每盘一个读线程，负责：

- 维护 `Q` 个 `POST_READ_SLOT` context
- 维护 `Q` 个 `READ_ACK` context
- 持续补满读 slot 深度
- 批量处理已完成的 `POST_READ_SLOT`
- 将读数据拷贝到 app medium
- 异步发出 `READ_ACK`

关键规则：

- `POST_READ_SLOT` 完成后，不能等 `READ_ACK` 返回再补投下一个 read slot
- `READ_ACK` 只负责回收 ACK context，不反向卡住 read slot 补投

### 2. 写平面

每盘一个写线程，负责：

- 维护 `Q` 个 `POST_WRITE_SLOT` context
- 维护 1 个独立 `WRITE_ACK_BATCH` flush context
- 持续补满写 slot 深度
- 批量处理已完成的 `POST_WRITE_SLOT`
- 将写数据应用到 app medium
- 生成并缓存 ACK range
- 独立 flush `WRITE_ACK_BATCH`

关键规则：

- `POST_WRITE_SLOT` 完成后，要先记录 ACK，再立即补投下一 write slot
- `WRITE_ACK_BATCH` 的发送与 slot 补投解耦
- 按当前约定，系统写请求完成仍以 ACK 成功为准

### 3. 完成事件处理模型

无论读平面还是写平面，都不能维持“每轮 wait 到一个事件，只处理一个完成”的模型。

需要改成：

- 每轮先尽量补投
- 再 wait
- 一旦有完成，尽量把当前已 signaled 的完成都 drain 掉
- 再回到补投阶段

也就是说，完成处理必须是“批量 drain”，不能只吃一个就回主循环。

## 为什么不继续坚持每盘一个线程

每盘一个线程虽然结构看起来简单，但会天然带来三个问题：

1. `POST_*_SLOT` 完成处理、数据拷贝、ACK 发送会互相串住
2. 单线程很难同时把队列补满、把完成吃干净、还把 ACK 路径跑顺
3. 深队列下吞吐更依赖“完成消费速度”，单线程容易成为每盘上限

所以这里不建议继续坚持“每盘只有一个 slot engine 线程”的模型。

## 为什么也不直接回退到旧模型

旧模型虽然快，但线程数量更多，状态更分散，取消和收尾路径更难统一。

当前更合适的方向是：

- 保留“按盘隔离”的新结构
- 保留当前已验证过的取消/清理思路
- 只把每盘数据面改成双平面

这样可以在不回退整体架构的前提下，把单盘吞吐恢复回来。

## 本阶段不动的部分

第一阶段先只动 App。

暂不主动调整：

- KMDF 会话/看门狗模型
- KMDF 当前 `POST_*_SLOT` 异步转发方式
- SCSI per-target queue 模型
- `WRITE_ACK_BATCH` 协议本身

原因是当前现象已经足够说明瓶颈主要在 App 每盘执行模型上，应先集中拆掉这里的串行点。

## 第二阶段候选优化

如果 App 数据面重建后，单盘吞吐仍明显低于预期，再考虑第二阶段：

- KMDF 侧为 slot 转发引入对象池
- 避免每个 slot 都做新的 `IRP / WorkItem / Buffer` 分配释放
- 进一步压低每个 slot 的固定转发开销

这部分是次级优化，不作为当前第一优先级。

## 落地步骤

1. 重写 App 每盘 slot engine 设计，拆成读平面线程和写平面线程
2. 为读平面引入独立 `READ_ACK` context 池，解除 read slot 与 ACK 的同步串行关系
3. 为写平面引入独立 ACK flush 路径，解除 write slot 与 ACK flush 的同步串行关系
4. 将完成处理从“单事件消费”改为“批量 drain”
5. 保持当前建盘、删盘、取消、退出清理逻辑语义不变
6. 完成后重新验证单盘 Q1/Q2/Q8/Q32 与多盘并发吞吐

## 验收预期

预期结果：

- 单盘顺序读写吞吐恢复到接近改造前水平
- 多盘总吞吐继续线性扩展
- 不引入新的 Q2/Q8/Q32 卡死
- 不破坏当前 `ct -> 压测 -> rm` 和退出清理闭环

## 当前建议

下一步直接从 App 开始重建，不先去怀疑全局锁，不先去改 SCSI，不先去改 KMDF。

当前最有价值的事情是：

- 把每盘单线程 `slot engine` 退役
- 改成每盘读/写双平面并发数据面
