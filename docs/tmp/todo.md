# 当前总目标

重建 `windows/tauri-client` 启动链与会话状态入口，去掉独立初始化页，收成“程序直接进入主页 + 会话状态按钮 + 启动期磁盘统一无效展示 + 会话成功后自动重扫与自连接”的最小闭环。

## 子步骤

1. 重建启动入口与前端编排边界
   - 重建 `App.vue`，删除独立初始化页渲染分支，程序启动后直接进入主页。
   - 保留当前窗口级快捷键拦截逻辑，例如禁用 `F5` / `Ctrl+R` 刷新。
   - 重建主页启动编排入口，明确“恢复配置、打开会话、启动重扫、自连接”四段流程的唯一调用链。
   - 将启动编排收口到主页专用最小组件 / 组合式模块，不把启动时序散落在多个 UI 组件回调中。

2. 重建 Tauri 命令边界，拆开“恢复配置”和“打开会话”
   - 重建当前 `initialize_client` 语义，拆成“只恢复配置 / runtime”和“只打开设备 / session”两个明确命令。
   - 保持 `queryHomeDiskList`、`rescanRuntimeDisks` 继续承接磁盘查询与重扫，不把它们并回初始化命令。
   - 保证命令职责单一：恢复配置不打开设备，打开会话不重复恢复配置。
   - 为前端返回稳定、明确、可文档化的会话状态结构，不新增含混中间态。

3. 重建主页启动期状态模型
   - 新增前端唯一会话状态源，至少区分：
     - `initializing`
     - `ready`
     - `failed`
   - 新增主页唯一磁盘展示阶段，至少区分：
     - 启动期展示阶段
     - 正常展示阶段
   - 严格区分“会话状态”和“磁盘真实状态”，不把“正在初始化 / 会话失败”写进 Backend 真实磁盘状态。
   - 所有启动期展示覆盖都只能在主页单一映射口完成，不允许散到 `DiskCard`、`DiskListPanel` 和 API 层各自补判断。

4. 重建主页启动顺序
   - 程序进入主页后，先恢复配置和磁盘列表 runtime。
   - 配置恢复完成后，先把所有磁盘展示为 `invalid`。
   - 在会话打开成功前：
     - 会话按钮显示初始化态；
     - 磁盘悬停提示显示“正在初始化”。
   - 会话打开成功后：
     - 会话按钮切到正常态；
     - 立即触发一次启动重扫；
     - 启动重扫成功后，正常磁盘从展示无效切回真实 `disconnected`；
     - 然后再进入自连接逻辑。
   - 会话打开失败后：
     - 会话按钮切到失败态；
     - 所有磁盘继续展示为 `invalid`；
     - 磁盘悬停提示统一改为“会话失败”。

5. 重建磁盘展示态映射
   - 保持当前磁盘真实状态类型不扩容，继续只使用现有真实状态模型。
   - 在主页增加唯一展示态映射：
     - 启动期统一覆盖为 `invalid + 正在初始化`
     - 会话失败统一覆盖为 `invalid + 会话失败`
     - 会话成功且重扫完成后，回退为 Backend 返回的真实状态
   - 保证内存盘、文件盘都走同一套展示覆盖逻辑，不为某一类盘单独开分支。
   - 保证重扫和自连接结束后，展示态完全回到真实状态，不留下“初始化残影”。

6. 重建会话状态按钮
   - 将当前顶部“会话正常”只读状态块重建为可点击按钮。
   - 保留现有视觉基调，只在必要范围内接管和覆写 `Element Plus` 按钮样式。
   - 初始化态：
     - 文案显示“正在初始化”
     - 绿圆替换为旋转指示
   - 正常态：
     - 显示绿圆 + “会话正常”
   - 失败态：
     - 显示红圆 + “会话失败”
   - 按钮点击统一打开会话状态对话框，不在按钮本体直接堆叠更多动作。

7. 重建会话状态对话框
   - 新增独立会话状态对话框，承接会话状态查看与失败重试。
   - 对话框内容视觉沿用当前初始化页骨架，但不再保留独立初始化页面。
   - 初始化态展示：
     - 旋转指示
     - “正在初始化”
     - 当前阶段说明文案
   - 正常态展示：
     - 成功状态图形 / 文案
     - 当前 `stateText`
     - 不显示重试按钮
   - 失败态展示：
     - 失败状态图形 / 文案
     - 错误说明
     - 显示重试按钮
   - 重试按钮只重试“打开会话 -> 启动重扫 -> 自连接”，不重复恢复配置，不重建主页。

8. 重建主页重扫语义与自连接触发点
   - 区分“启动重扫”和“用户手动重扫”。
   - 启动重扫属于启动编排内部步骤，不弹额外成功提示，不打断启动链。
   - 用户手动重扫继续保留现有按钮入口和交互语义。
   - 自连接严格放在“启动重扫成功之后”，不允许在磁盘还处于启动期展示无效时提前连接。
   - 保证托盘唤起主窗口时不重复执行启动自连接。

9. 重建错误边界与最小验收口径
   - 如果恢复配置失败，主页仍应稳定进入，并展示明确错误，不回退到旧初始化页。
   - 如果会话打开失败，主页仍可见，磁盘列表仍可见，但统一处于会话失败覆盖态。
   - 如果启动重扫失败，会话按钮可正常显示失败 / 异常态，磁盘列表不切回旧初始化页。
   - 会话状态对话框中的重试，是当前阶段唯一正式的会话恢复入口。
   - 当前阶段验收只覆盖：
     - 程序启动直接进入主页
     - 启动期全部磁盘显示无效
     - 会话成功后自动重扫并恢复真实未连接状态
     - 会话失败后顶部按钮和悬停提示正确
     - 失败后点击状态按钮可打开对话框并重试

10. 同步文档与归档
    - 完成每一子步骤后，立即归档到 `docs/progress/YYYY-MM-DD.md`。
    - 如果会话状态按钮、启动链或展示态边界形成长期规则，同步更新正式 UI / 架构文档。
    - 本轮 `todo` 始终只保留当前未完成目标，不回填已完成内容。

## 当前轮边界

- 当前轮只收 `windows/tauri-client` 的启动链、主页会话状态入口、会话状态对话框和启动期磁盘展示逻辑。
- 当前轮坚持“重建唯一正确实现”，不保留旧初始化页与主页双轨启动逻辑。
- 当前轮不为运行期 `session broken` 事件、自愈恢复、后台自动重连建立新体系。
- 当前轮不新增 Backend 真实磁盘状态类型，不把“正在初始化 / 会话失败”写入真实盘状态。
- 当前轮不重构整套全局 store，只在现有前端结构中收紧唯一启动编排入口和唯一展示映射口。
- 当前轮不扩展设置页、不扩展托盘行为、不新增额外会话控制面板。
- 当前轮 UI 仍遵循 `Element Plus` 为基础的规则，只在实现目标视觉和交互语义时做必要接管。

## 当前唯一下一步

重建主页启动顺序，把“恢复配置 -> 启动期统一无效 -> 打开会话 -> 启动重扫 -> 恢复真实状态 -> 自连接”收成唯一链路，并保证会话失败时继续停留在统一失败覆盖态。
