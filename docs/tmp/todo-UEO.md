# `tauri-client` 重扫体验优化清单

## 0. 当前范围

本清单只服务于 `windows/tauri-client` 的“重扫运行态磁盘”体验优化。

这里固定讨论：

- 手动点击主页“重扫”按钮后的执行体验
- `tauri` 后端重扫命令的执行模型
- 本地盘与网络盘的分阶段提交
- 重扫期间的前端状态、事件通知和并发约束

这里明确不做：

- `NetworkMedia` 读写数据面改造
- 网络协议改造
- `cache` 接线
- 磁盘读写性能优化

## 1. 当前现状

### 1.1 当前命令是整段同步执行

当前 `rescan_runtime_disks` command 仍是同步路径：

- 先拿住 `disk_catalog` 锁
- 再执行 `sync_runtime_state`
- 再执行本地盘重扫
- 再执行网络盘重扫
- 最后保存 `client_state`
- 整段完成后才把最终 snapshot 返回给前端

这意味着：

- 只要网络盘阶段慢，整个 command 返回就慢
- `disk_catalog` 锁被长时间持有
- 页面虽然是 `await invoke()`，但宿主侧长任务会让桌面程序出现明显卡顿，甚至“未响应”

### 1.2 网络盘重扫虽然已收成四段流水，但仍是一次性返回

当前网络盘重扫主线已经按下面四段组织：

1. 收集全部网络盘任务
2. 按 `server_addr` 汇总分组
3. 获取 candidate
4. 统一裁决并统一提交

但这条流水当前仍然包在一次同步 command 里跑完。

因此它虽然语义上已经收口，但体验上还是：

- 前端只能等整轮结束
- 不能做到“哪个网络盘先完成，哪个先更新”
- 不能把网络慢阶段藏到后台

### 1.3 前端当前没有独立的“重扫中”状态

当前前端只是在调用 `rescan_runtime_disks()` 后等待最终结果：

- 没有独立的 `rescanLoading`
- 重扫按钮没有正式 loading 态
- 手动重扫仍复用列表级 `loading`
- 当前列表刷新模型仍偏向“等最终 snapshot 再整体替换”

这和我们要的体验不一致。

### 1.4 当前已经有可复用的事件刷新基础

当前后端已经有 `network-runtime-changed` 事件轮询与前端监听链。

这说明：

- 不需要为了这次优化重建整套前后端通知体系
- 可以复用现有“后端发事件，前端补 query”的方向
- 真正需要改的是“重扫任务如何后台化、如何分阶段提交、如何表达任务生命周期”

## 2. 优化目标

本轮目标固定收成下面几件事：

1. 手动点击“重扫”后，窗口不再卡死，不再把整个 `tauri-client` 拖成未响应
2. 重扫按钮进入明确 loading 态，并阻止重复触发
3. 内存盘和文件盘沿用现有逻辑，完成本地阶段后立刻更新
4. 网络盘在后台继续重扫，哪个分组先完成就先提交哪个；未完成前保持旧状态
5. 现有网络盘冲突裁决、mounted 保持规则、invalid 规则不改变
6. 重扫过程中不引入第二套长期 runtime 状态源，不制造状态分叉

## 3. 固定边界

### 3.1 手动重扫和启动重扫共用一套模型

本轮主目标虽然先解决主页手动点击“重扫”的卡顿问题，但启动链里的：

- `open_session`
- `rescan_runtime_disks`
- `auto_mount`

也必须并入同一套重扫模型。

固定口径：

- 启动流程一旦进入 `rescan_runtime_disks`，重扫按钮也进入 loading
- 启动阶段重扫完成后，重扫按钮恢复正常
- 不为启动链另起第二套“隐藏重扫状态”
- 手动重扫和启动重扫只区分触发来源，不区分底层任务模型

### 3.2 不新增磁盘级过渡状态

首版不为磁盘新增类似：

- `rescanning`
- `refreshing`
- `pending`

这类新的长期展示状态。

未完成的网络盘固定保持旧状态，直到新结果真正提交。

原因很直接：

- 这更符合极简核心原则
- 也避免前后端额外维护第二套状态机

### 3.3 不复制一份完整 runtime 快照长期维护

重扫过程中允许有临时的任务数据、candidate 数据、裁决结果。

但这些都只能是本轮任务内部的临时数据，不能变成新的长期状态表。

固定口径：

- `DiskRuntimeStore` 仍是唯一运行态真状态
- 前端磁盘列表仍只来自正式 query 结果
- 不做“前端一份 pending list + 后端一份 runtime_store”双轨维护

### 3.4 网络盘仍保持四段语义

网络盘重扫仍然保持当前已收好的四段语义：

1. 收集任务
2. 分组
3. 获取 candidate
4. 裁决并提交

允许调整的是执行模型与提交粒度，不允许把已经收好的裁决语义再打散回旧的边走边写。

### 3.5 同时只允许一个 active rescan

手动重扫不能并发叠多轮。

固定口径：

- 同一时刻最多一个 active rescan job
- 再次点击时直接拒绝或复用当前 job
- 不允许多个后台重扫同时改 `runtime_store`

### 3.6 现有 watcher 不能和重扫后台线程打架

当前已有网络事件 watcher 会周期执行 `sync_runtime_state`。

引入后台重扫后，必须统一协调下面这两类动作：

- 后台重扫任务
- 周期 watcher 同步

不允许它们在没有约束的情况下同时推进同一批运行态修改。

## 4. 需要完成的工作

### 4.1 收一个最小的 rescan coordinator

后端需要有一个最小协调器，负责：

- 启动后台重扫
- 记录当前是否已有 active job
- 表达 job 的开始、完成、失败
- 防止重复启动

这里不要一开始就做复杂任务系统。

最小闭环就够：

- idle
- running
- last_error 或一次性失败通知

### 4.2 把当前同步 command 改成“触发后台任务”

当前 `rescan_runtime_disks` 不应继续承担“整段跑完再返回”的职责。

它需要改成：

- 负责启动一轮后台重扫
- 尽快返回
- 让前端进入按钮 loading

如果当前已有 active rescan，则直接返回“正在进行中”的明确信号。

### 4.3 把重扫执行拆成分阶段提交

当前整体串行路径要拆成下面两段：

1. 本地阶段
2. 网络阶段

本地阶段要求：

- 先执行 `sync_runtime_state`
- 再执行本地盘重扫
- 立刻提交到 `runtime_store`
- 立刻保存状态
- 立刻通知前端刷新

网络阶段要求：

- 在后台继续跑网络盘四段流水
- 至少按 `server_addr` 分组提交
- 每完成一个提交批次，就保存状态并通知前端刷新

这样可以满足：

- 内存盘和文件盘先更新
- 网络盘后续逐步更新

### 4.4 把锁范围压缩到短临界区

当前最大问题之一，是长时间持有 `disk_catalog` 锁跨越网络 I/O。

需要改成：

- 收集任务时短暂持锁
- 提交结果时短暂持锁
- 保存状态时在明确边界做

不再允许：

- 整个网络盘 `connect/auth/open/describe` 期间持续持有 `disk_catalog`

### 4.5 明确后台事件模型

首版不必做复杂进度条，但至少要把任务生命周期补齐。

建议最小事件集合：

- rescan started
- local stage committed
- network batch committed
- rescan finished
- rescan failed

前端可以继续复用：

- 收到事件后重新 query 当前正式列表

按钮 loading 的结束则不能只靠 `network-runtime-changed`，还需要明确的“任务结束”信号。

### 4.6 前端补独立的重扫状态

前端需要新增独立的 `rescanLoading`，而不是继续复用全局 `loading`。

固定目标：

- 手动重扫时只让重扫按钮 loading
- 启动链进入 `rescan_runtime_disks` 后，重扫按钮同样 loading
- 不把整张列表重新切成 skeleton
- 当前列表在重扫期间继续显示
- 收到后端阶段提交事件后再局部体现结果

这里的正式口径应当是：

- `rescanLoading` 表示“当前存在 active rescan job”
- 它不区分这轮 job 是用户手动触发，还是启动链触发
- 如果后续确实需要区分来源，只额外补一个轻量 `source` 字段，不再拆成两套 loading 状态

### 4.7 明确手动重扫期间的其他交互策略

一旦重扫变成后台任务，之前“页面卡住所以用户点不了别的”的隐式串行就没了。

因此必须明确下面这些操作在重扫期间怎么处理：

- mount
- eject
- create
- update
- delete
- 网络 draft 相关操作

这里固定收成统一临时禁用策略。

正式口径：

- 重扫期间禁止再次重扫
- 重扫期间不允许挂载
- 重扫期间不允许拔出
- 重扫期间不允许编辑
- 重扫期间不允许删除
- 重扫期间不允许新增
- 重扫期间不允许网络 draft 相关操作

实现方式优先收成前端和命令入口双闸口：

- 前端直接禁用对应按钮和入口
- 后端命令入口也要拒绝，不能只靠前端约束

这样可以避免后台重扫期间出现运行态并发改动。

### 4.8 区分“手动重扫”和“启动重扫”的消费方式

当前启动链会在打开会话后等待重扫结果，再进入自动挂载。

后台化后，要明确两类消费方式：

- 手动重扫：触发后立即返回，由事件驱动刷新
- 启动重扫：可以复用同一套后台实现，但启动链仍可等待 job 完成

这样可以避免：

- 为了启动链，再保留一套旧同步逻辑
- 或者为了手动重扫，把启动链强行改成不受控后台漂移

同时还要明确启动阶段的 UI 口径：

- 启动链进入重扫后，按钮直接显示 loading
- 启动链等待 job 完成期间，按钮保持 loading
- 自动挂载开始前，重扫必须已经完成
- 重扫完成或失败后，按钮再恢复

## 5. 验收口径

本轮完成后，至少要满足下面这些结果：

1. 手动点击重扫后，窗口保持可响应
2. 重扫按钮进入 loading，重复点击不会叠多轮任务
3. 本地盘阶段完成后，内存盘和文件盘立刻更新
4. 网络盘在后台按批次逐步更新，未完成项保持旧状态
5. 同 backend 冲突裁决、mounted 保持、invalid 收口语义不回退
6. 启动链进入重扫阶段时，按钮同样进入 loading，并在该轮结束后恢复
7. 重扫期间挂载、拔出、编辑、删除、新增和网络 draft 操作全部临时禁用
8. 重扫结束后，按钮 loading 正常结束
9. 某个网络分组失败时，不回滚已经提交成功的其他分组结果

## 6. 当前不做

本轮明确不做下面这些扩展：

- 磁盘级实时进度条
- 每个网络盘单独百分比
- 多轮重扫排队
- 复杂任务中心
- 前端维护一份独立 pending 列表
- 为重扫再造一套新的运行态模型

## 7. 实施顺序建议

建议按最小闭环推进：

1. 先补后端 `rescan coordinator` 和单 active job 闸口
2. 再把本地阶段与网络阶段拆开，并接通事件通知
3. 再补前端 `rescanLoading`、按钮 loading 和刷新链
4. 最后补重扫期间其他交互的并发约束与验收测试

这样收，复杂度最低，也最符合当前开发原则第一条。
