  第四章 · SMB 为什么不行（5:00 - 7:30）
═══════════════════════════════════════════════════════════════════════════

【画面：动画。一台NAS服务器 -> 内网 -> 路由器 -> 公网 -> 一台笔记本电脑。内网到路由器之间显示绿色通畅。路由器到公网之间的SMB端口标红，显示"端口被封"]

SMB——Server Message Block——Windows文件共享的标准协议。

它在局域网里，确实很好用。映射一个网络驱动器，你的电脑上多出一个Z盘，文件就在那里，软件也能直接访问。

但是——请注意这个"但是"——

它只在局域网里好用。

一旦你想从外面访问——比如你在公司，想连回家里的NAS——问题就来了。

【画面：逐条展示问题，配合红色叉号]

第一个问题：端口被封。

SMB走的是固定端口。135、137、139、445——这些端口，运营商基本都会封掉。

为什么？因为这些端口历史上被大量恶意软件利用。勒索病毒、蠕虫，最爱走的就是SMB端口。

所以你的ISP，不管是宽带还是移动网络，出于安全考虑，把这些端口全给你封了。

【画面：端口列表一个个被打上红色"封禁"章]

你有公网IP？没用。
你有IPv6？没用。
端口不通，一切白搭。

【弹幕预判：（已阅 确实被封了）（我之前试过 根本连不上）（原来是被运营商封了 我说怎么不行）】

好，你说——"那我不用标准端口，我把SMB强制代理到其他端口上，行不行？"

理论上可以。

实际操作起来——端口冲突、代理失败、认证超时、各种玄学问题，层出不穷。

而且最关键的是——SMB不止一个端口。你根本不知道除了445，还有哪些端口要跟着一起代理。可能你代理了445，结果认证走了139，数据传输又走了另一个端口——你完全控制不了。

【画面：一张迷宫图，标注"SMB端口代理的日常"各条路径相互纠缠]

结论：SMB跨公网，此方案基本无解。

  第五章 · iSCSI 为什么也不行（7:30 - 10:00）
═══════════════════════════════════════════════════════════════════════════

【画面：切回UP主，端起水杯喝一口水，示意"我们进入下一段硬核内容"]

好，你是一个硬核玩家。你说——"SMB不行，那我用iSCSI总行了吧？"

iSCSI —— Internet Small Computer System Interface。

跟SMB不一样，iSCSI提供的是块设备——block device——而不是文件系统。它比SMB更底层。

你的电脑通过iSCSI连上一块远程硬盘，Windows看到的就是一块"物理磁盘"。你可以给它分区、格式化，就像插了一块新硬盘一样。

听起来很完美，对吧？

【画面：动画。iSCSI服务器 -> 内网 -> 路由器 -> 公网 -> 我的电脑】

但iSCSI有两个致命问题。

问题一：门户IP暴露。

【画面：屏幕录制模拟。iSCSI发现门户的过程中，服务器返回了一个内网IP地址：192.168.1.100。客户端在外网，收到这个地址后一脸茫然】

iSCSI在发现阶段——叫"门户发现"——服务器会返回它的IP地址给客户端。

但如果你的iSCSI服务器在内网——192.168.1.100这种地址——它就会把这个内网地址，原封不动地告诉外网的客户端。

你人在星巴克，连上家里的iSCSI，收到一个"192.168.1.100"。

请问，你去哪找192.168.1.100这个主机？

【画面：UP主摊手，表情无奈。配文字："我上哪找去？"】

【弹幕预判：（绷不住了）（这不就是那个经典问题吗——我连上了 但是连到哪了？）（内网穿透的终极难题）】

问题二：拆包粘包零容忍。

【画面：动画演示。一个iSCSI数据包在传输过程中被中间设备分割成两个包，接收端直接报错]

iSCSI对网络环境极其敏感。它在传输过程中，不允许任何中间节点对数据包做拆包、粘包、重组。

但是跨公网传输，数据包经过NAT、经过多层路由、经过各种中间设备——拆包粘包几乎是不可避免的。

而iSCSI遇到这种情况——直接拒绝。没有商量。

所以，跨网段传输？pass。
内网穿透方案？全pass。
任何中间有NAT的场景？一律pass。

【画面：一张表，两行对比】
  SMB跨公网：端口被封 → 无解
  iSCSI跨公网：IP暴露 + 拆包零容忍 → 无解

【画面：切回UP主。语气稍作调整，进入"稍微硬核一点"的部分]

好，到这里，你可能要问了——你是怎么做到的？

既然SMB不行、iSCSI也不行，那YumeDisk凭什么行？

【画面：架构图动画，一层一层展开]

YumeDisk的架构，分了这么几层：

最底层，是两个Windows内核驱动——一个KMDF控制驱动，一个StorPort存储端口驱动。这两个驱动负责跟Windows的存储栈打交道——让Windows以为，这里真的插着一块物理硬盘。

你要知道，Windows对磁盘的管理是非常严格的。不是你写一个用户态程序就能让系统认出一个新磁盘来。你必须走正式的驱动层——SCSI命令、存储端口初始化、总线枚举——全套流程。

YumeDisk选择了正面打通这一层，而不是绕开它。

【画面：架构图继续展开。驱动之上是AppKernel——一个用户态数据面内核]

驱动之上，是AppKernel——用户态数据面内核。它负责会话管理、磁盘IO调度、心跳维持——是整个系统的"发动机"。

AppKernel之上，是BackendRust——用Rust写的一个核心运行时。它管理着每一块磁盘的生命周期，协调不同类型的存储介质。

再往上，是网络协议层——一个自己设计的、专为跨网络块传输优化的轻量协议。没有SMB的历史包袱，没有iSCSI的僵化约束。

【画面：网络协议对比。传统SMB胖协议 vs YumeDisk轻量协议，包大小、握手次数、端口灵活性的对比]

这个协议端口灵活、传输稳定、部署简单。32KB的块大小，在一条请求里就能完成一次读写——不需要几十次握手。

最上层，是Tauri桌面客户端——Vue 3 + Element Plus的前端，Rust后端，打包成一个轻量的Windows桌面应用。系统托盘常驻，开机自启，不占资源。

【画面：整张架构图全景]
  ┌─────────────────────────────────────┐
  │  Tauri桌面客户端 (Vue 3 + Rust)     │
  ├─────────────────────────────────────┤
  │  BackendRust (核心运行时)            │
  ├─────────────────────────────────────┤
  │  network-core (轻量跨网络协议)       │
  ├─────────────────────────────────────┤
  │  AppKernel (数据面内核)              │
  ├─────────────────────────────────────┤
  │  YumeDiskKMDF + YumeDiskSCSI (驱动)  │
  ├─────────────────────────────────────┤
  │  Windows 存储栈                      │
  └─────────────────────────────────────┘

服务端呢？Go写的——一个gateway负责接受客户端连接和路由，一个storer负责实际的存储数据面。简单、高效。不需要装TrueNAS，不需要配RAID，一个可执行文件就能跑起来。

而且YumeDisk不只支持网络存储。

它目前支持三种介质——

第一种：内存盘。把你的内存变成一块超高速磁盘。适合临时缓存、编译临时目录、跑分测试。读写速度就是你的内存带宽——几十GB/s。

第二种：文件盘。把一个本地文件——可以是RAW、VMDK、VHD——直接挂载为一块磁盘。即开即用，不依赖虚拟机软件。

第三种：网络盘。这就是我们今天的主角——远端存储，通过网络协议变成你电脑上的本地磁盘。

三种介质，同一种挂载方式，同一套管理体验。
