# 开发文档

## 1. 当前开发口径

当前阶段不再以旧 `WAIT_EVENT` 原型为目标。开发口径要收束到最终多盘 app-owned media queue：

- `YumeDiskKMDF` 负责单一 session、watchdog、direct I/O buffer 和异步 slot transport。
- `YumeDiskSCSI` 负责 per-target read/write queue、`eventId`、write `seq`、ACK bitmap 和 SRB completion。
- `RWTestApp` 负责 per-disk medium、per-disk read/write workers、read/write slot 预投和独立 `WRITE_ACK_BATCH` flush。

早期 `MyKWDF3 -> MyCtl`、`MyDisk2 -> MyDisk`、`MyApp5 -> MyApp` 的改造路线只作为历史背景；当前开发、排查和验收统一使用 `YumeDiskKMDF`、`YumeDiskSCSI`、`RWTestApp`。

## 2. 目标实现边界

以下内容是下一阶段多盘并发重建必须落到的唯一边界，后续不回到旧分支：

- App 不直接访问 `GUID_DEVINTERFACE_STORAGEPORT`，只打开 KMDF 自定义设备接口。
- KMDF 持有与 App file object 同生命周期的 miniport `HANDLE + FILE_OBJECT + DEVICE_OBJECT`。
- App file cleanup/watchdog 超时统一走 `RemoveAllDisks | SessionClose -> drain in-flight IO -> close miniport handle`。
- 数据面只保留 `POST_READ_SLOT`、`POST_WRITE_SLOT`、`READ_ACK`、`WRITE_ACK_BATCH`。
- `POST_WRITE_SLOT` 只提交 write slot，不携带 ACK trailer。
- `WRITE_ACK_BATCH` 是唯一写确认入口，并支持逐项失败结果。
- SCSI read/write 队列必须按 target 下沉，不回到 adapter 全局数据面锁。
- App 侧每个磁盘使用独立的小常数 worker 组，queue depth 按盘解释。
- hot path print 只用于短期定位，做性能采集前必须关闭。

## 3. 下一目标停止线

下一阶段工作应围绕最终多盘效果推进，而不是重新设计旧原型：

1. 按最终模型维持多盘并发数据面：`KMDF session -> SCSI per-target queue -> App per-disk workers`。
2. 接通 `建盘 -> 枚举 -> probe read -> steady-state read/write -> Ctrl+C / RemoveAllDisks / App 退出`。
3. 跑通 `Q1T1 / Q8 / Q32` 手工压测矩阵。
4. 记录吞吐、kernel CPU、ACK 失败计数和 cleanup 稳定性。
5. 验证多盘并发没有 adapter 级全局队列串扰、App 全局 worker pool 串扰、pending session/slot/SRB 残留。
6. 将已验证事实归档到 `docs/progress/`，长期规则再同步到正式文档。

当前轮具体任务仍以 `docs/tmp/todo.md` 为准；该文件只承接当前待办，不在正式文档中复制已完成项。

## 4. 组件开发约束

## 4.1 YumeDiskKMDF

必须保持的约束：

- session 与 App file object 绑定。
- watchdog 只在 KMDF 内维护，不转发给 SCSI。
- miniport handle 不能退回 device-global lazy cache 作为数据面默认路径。
- `POST_READ_SLOT` / `POST_WRITE_SLOT` 正常路径返回 pending，由异步 completion 完成原始 App request。
- cleanup 必须等待或取消 in-flight slot transport 后再释放 session 资源。
- locked session 后续 IOCTL 直接失败，不能被 heartbeat 或新 slot 救活。
- `SessionLock` 只能用于低频生命周期路径，不能作为每个 slot 的高频准入锁。
- submit queue / free list 的锁内只做链表移动和小字段更新。
- `IoCallDriver`、`WdfRequestComplete`、IRP 复用初始化、对象分配释放都必须在锁外。
- 热路径优化按“激进更新原则”重建，不在旧 per-slot transport 上长期叠补丁。
- 新 session-owned transport runtime 落地后，旧 per-slot IRP/work item 路径和旧每 slot 获取 `SessionLock` 的准入路径必须删除，不保留双轨。

需要重点观察：

- 同步完成和异步 pending 都必须正确完成原始 App `WDFREQUEST`。
- completion 路径释放 session 引用，不能在 cleanup 中提前释放。
- `STATUS_IO_DEVICE_ERROR` 等 miniport 中断类错误要映射到更接近取消/清理语义的状态，避免 App 误判磁盘永久坏死。
- cleanup/watchdog 不能持有 `SessionLock` 等待 `PendingSlotZeroEvent`。
- 对象池满时初版应返回 busy 让 App 重投，不在 KMDF 内无限等待池对象。

## 4.2 YumeDiskSCSI

必须保持的约束：

- 每个 target 自己维护 read/write queue。
- `PostedReadSlots`、`PendingReads`、`PostedWriteSlots`、`PendingWrites` 不回到 adapter 全局共享链表。
- ACK 落账在 write queue 状态下完成，但大数据 copy 和 SRB completion 在锁外。
- `WRITE_ACK_BATCH` 先完成 ACK 控制请求回 App，再完成满足条件的系统 WRITE SRB。
- session close、disk remove、reset、cancel 路径都必须完成或取消 pending SRB。
- per-target queue 继续使用 `KSPIN_LOCK`，不改成 wait lock。
- `KSPIN_LOCK` 内只做队列摘取、计数、bitmap、状态位更新。
- `RtlCopyMemory`、`StorPortNotification(RequestComplete)`、`DiskFree` 都必须在锁外。
- cleanup/remove 必须先把 pending requests / posted slots 移到本地 list，再在锁外完成和释放。

需要重点观察：

- Q8/Q32 下是否还会出现 slot 已完成给 App、原始 SRB 丢失引用的情况。
- stale/cancelled ACK 是否只返回逐项失败，不拖死整条队列。
- read probe 是否稳定完成，避免“设备有了但盘枚举不出来”。
- 如果锁内线性查找成为热点，优先增加 ready list、pending-unissued list 或 eventId index，不扩大临界区。

## 4.3 RWTestApp

必须保持的约束：

- App 单实例。
- 驱动包和设备实例启动收敛在连接 KMDF 前完成。
- 每盘独立 medium、worker 组和 queue depth。
- 不回到跨盘共享的全局 read worker pool、write worker pool、ack flush thread 调度语义。
- `POST_WRITE_SLOT` 不捎带 ACK。
- `WRITE_ACK_BATCH` 逐项失败要打印并计数。
- 系统取消只追到 `SCSI`；App 不维护额外的全链路取消协议。

需要重点观察：

- `queueDepth <= 32` 的限制与 `WaitForMultipleObjects` handle 上限一致。
- 每盘 worker 数量是否已经足够撑满单盘吞吐，而不会把上下文切换重新拉爆。
- cleanup 时 pending overlapped 请求是否都被取消或完成。

## 5. 验证清单

### 基础控制面

- App 单实例。
- 驱动包缺失自动安装。
- 设备实例缺失自动创建。
- 重复设备自动清理。
- `QUERY_INFO` 能确认目标 miniport。
- 单盘创建和删除。
- 多盘创建和删除。
- App 正常退出删盘。
- App 崩溃或 heartbeat 超时删盘。

### 数据面

- 系统 probe read 能完成。
- 小块读写。
- 1M 顺序读写。
- `Q1T1` 顺序读写。
- `Q8` 顺序读写。
- `Q32` 顺序读写。
- 多盘并发 `Q1T1`。
- 多盘并发高 QD。

### 异常路径

- Ctrl+C。
- `RemoveAllDisks`。
- 删除盘时存在 pending read/write。
- session close 时存在 pending posted slots。
- stale/cancelled `WRITE_ACK_BATCH`。
- miniport reset/remove。
- App 停止 heartbeat。

### 锁与并发回归

- 高 QD 下 `POST_*_SLOT` 不因 `SessionLock` 竞争导致 kernel CPU 异常升高。
- cleanup/watchdog 不持有 session 锁等待 pending slot drain。
- SCSI read/write drain 的 copy 和 completion 都发生在 queue lock 外。
- `WRITE_ACK_BATCH` 完成原始 WRITE SRB 前已经释放 `WriteQueueLock`。
- remove/cleanup 时 pending 对象先移动到本地 list，再锁外完成。
- 锁内线性查找、bitmap range ACK 的最长耗时有 debug counter 或采样手段可定位。
