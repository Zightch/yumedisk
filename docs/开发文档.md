# 开发文档

## 1. 开发目标

本阶段的目标不是一次性做完全部优化，而是先完成一个可跑通的原型，满足以下条件：

- App 启动时能自动完成驱动与设备自修复
- App 全局单实例
- App 打开时可建盘
- App 退出时必删盘
- 系统读写能通过事件转发到 App
- 多盘按多 Target 工作

## 2. 项目改造映射

## 2.1 MyKWDF3 -> MyCtl

当前 `MyKWDF3` 中最有价值的是控制设备框架：

- 文件对象回调
- 自定义设备接口 GUID
- 默认控制队列

当前 `MyKWDF3` 中与最终方案不一致的部分：

- 子设备模型
- 控制设备和子设备之间的读写桥接逻辑

改造方向：

- 保留控制设备主框架
- 移除当前子设备读写桥接逻辑
- 改为 storageport 代理型控制驱动

建议重点文件：

- `MyKWDF3/MyKWDF3/main.c`
- `MyKWDF3/MyKWDF3/CtrlDev.c`
- `MyKWDF3/MyKWDF3/CtrlDev.h`
- `MyKWDF3/MyKWDF3/define.h`
- `MyKWDF3/MyKWDF3/Utils.*`

## 2.2 MyDisk2 -> MyDisk

当前 `MyDisk2` 已具备以下基础：

- Storport miniport骨架
- 多盘方向的 Target 解析
- `REPORT_LUNS`
- `INQUIRY`
- `READ_CAPACITY`
- `READ_CAPACITY16`
- 普通 `READ/WRITE`

需要新增的核心能力：

- 私有控制命令分发
- session 状态
- 磁盘 presence 状态
- 事件等待队列
- 事务表
- 读写挂起和回包完成
- 会话丢失时删盘

建议重点文件：

- `MyDisk2/MyDisk2/main.c`
- `MyDisk2/MyDisk2/disk.c`
- `MyDisk2/MyDisk2/disk.h`
- `MyDisk2/MyDisk2/define.h`

## 2.3 MyApp5 -> MyApp

当前 `MyApp5` 已具备：

- GUID 枚举能力
- 打开设备接口能力
- 同步和异步 `DeviceIoControl` 测试能力

需要改造：

- 增加单实例控制
- 增加驱动安装器
- 增加设备实例扫描器
- 增加设备实例创建器
- 增加重复设备清理器
- 不再直连 storage 类或旧测试设备
- 统一改为连接 `MyCtl`
- 增加心跳线程
- 增加 `WAIT_EVENT` 池
- 增加事务处理器
- 增加内存磁盘模型

建议重点文件：

- `MyApp5/MyAppCtrl/main.cpp`
- `MyApp5/MyAppDev/main.cpp`
- `MyApp5/utils/utils.*`
- `MyApp5/CMakeLists.txt`

## 3. 分阶段实施

## 阶段1：MyCtl最小化落地

目标：

- App 能保证单实例
- App 能保证 `MyCtl` 和 `MyDisk` 驱动包已安装
- App 能保证 `MyCtl` 和 `MyDisk` 设备实例存在且唯一
- App 能枚举到 `MyCtl`
- App 独占打开 `MyCtl`
- `MyCtl` 能在内核中找到 `MyDisk`
- `MyCtl` 能转发 `QUERY_INFO`

完成标准：

- 在未安装驱动的机器上，App 可完成安装
- 在设备实例缺失时，App 可自动创建设备
- 在存在重复设备时，App 可自动清理到唯一实例
- `MyApp` 能成功建立会话
- `QUERY_INFO` 返回固定签名和版本号

## 阶段2：会话与删盘链路

目标：

- App 断开时，`MyCtl` 能触发 `REMOVE_ALL_DISKS`
- `MyDisk` 收到命令后能删除全部盘并通知总线变化

完成标准：

- `CreateFile(MyCtl)` 后建盘可见
- `CloseHandle(MyCtl)` 后所有盘都消失

## 阶段3：WAIT_EVENT 骨架

目标：

- `MyCtl` 支持挂起多个 `WAIT_EVENT`
- `MyDisk` 支持等待队列
- `MyApp` 能维持 64 深度

完成标准：

- 无事件时 `WAIT_EVENT` 可稳定挂起
- 有事件时 `WAIT_EVENT` 可按顺序完成

## 阶段4：写请求闭环

目标：

- 系统写盘进入 `WRITE_REQ`
- App 收到写事件
- App 应答 `WRITE_ACK`
- `MyDisk` 完成原始写请求

完成标准：

- 普通文件写入路径可闭环

## 阶段5：读请求闭环

目标：

- 系统读盘进入 `READ_REQ`
- App 收到读事件
- App 回送 `READ_REPLY`
- `MyDisk` 完成原始读请求

完成标准：

- 普通文件读取路径可闭环

## 阶段6：多盘与异常处理

目标：

- 多 Target 稳定工作
- 删除单盘稳定
- 会话丢失时全盘删除
- 事务超时可恢复

## 5.2 首批验证点

优先验证以下最小链路：

1. 未安装环境下启动 `MyApp`
2. `MyApp` 自动安装驱动包
3. `MyApp` 自动创建唯一 `MyCtl`/`MyDisk` 设备实例
4. `MyApp` 打开 `MyCtl`
5. `MyCtl` 能找到 `MyDisk`
6. `QUERY_INFO` 成功
7. `CREATE_DISK` 成功
8. 磁盘管理器能看到盘
9. 关闭 `MyApp`
10. 盘立即消失

## 7. 编码顺序建议

建议严格按以下顺序推进，不要先做完整数据面：

1. App 单实例与自安装
2. `MyCtl` 会话入口
3. `MyDisk` 私有命令入口
4. 建盘/删盘
5. `WAIT_EVENT` 队列
6. 写路径
7. 读路径
8. 超时和异常恢复

## 8. 当前已确定的设计结论

- 多盘用多 Target，不用多 LUN
- `REPORT_LUNS` 只反映单 Target 下的 LUN 0
- App 不在线时必须无盘
- App 启动时必须收敛驱动包与设备实例到唯一形态
- 删盘按激进热拔出处理
- `MyCtl` 是伴随 KMDF 控制驱动
- `MyDisk` 是 Storport miniport
- `MyApp` 只连 `MyCtl`
