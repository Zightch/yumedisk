## 4. 交互式终端用法

### 4.1 适用场景

- `execute_pwsh` 适合一次性命令：执行完即返回结果，不保留进程上下文。
- `terminal_*` 适合需要持续存在的会话：启动长期运行程序、持续看日志、分多步输入命令、等待异步输出。
- 联调中如果要在虚拟机里持续跑 `gateway`、`storer`、`rust-cli`、测试程序或观察实时日志，优先使用 `terminal_*`。

### 4.2 基本流程

1. `terminal_start`
   - 创建一个 PowerShell 交互终端。
   - 可以只起一个空终端，再后续输入命令。
   - 也可以在创建时直接附带 `command` 或 `program + program_args` 启动目标进程。
2. `terminal_write`
   - 向该终端标准输入写入命令或交互内容。
   - 适合分步执行，例如先切目录，再启动程序，再输入控制命令。
3. `terminal_read`
   - 读取终端输出。
   - 联调时通常用 `incremental` 或 `lines` 方式持续拉取新增输出，避免重复读取整段历史。
4. `terminal_status`
   - 查看终端当前是否还存活、子进程是否仍在运行。
5. `terminal_close`
   - 关闭终端，必要时用 `kill=true` 一并终止其子进程。
6. `terminal_list`
   - 查看当前机器上还存在哪些已创建的终端会话，适合回收遗留进程。

### 4.3 使用建议

- 需要拿一次结果时，用 `execute_pwsh`，不要为了单条命令额外起交互终端。
- 需要后台持续运行时，用 `terminal_start` 创建独立会话，并记录 `session_id`。
- 启动服务类程序后，不要假设命令已经退出，要通过 `terminal_read` 和 `terminal_status` 确认实际状态。
- 如果一个验证场景会同时跑多个进程，建议一个进程对应一个终端，避免多个程序输出混在同一终端里。
- 结束联调后要主动 `terminal_close`，避免虚拟机里遗留后台进程影响下一轮验证。

### 4.4 当前联调中的典型用途

- 在 `vm_win10` 或 `vm_win11` 中长期运行 `gateway`、`storer`、`rust-cli`、原生测试程序。
- 在故障路径测试中，一边保持服务运行，一边分步注入停止、断链、替换驱动、重新连接等动作。
- 在程序无窗口或日志只走标准输出时，持续读取运行日志。

## 5. 使用约束

- 非必需不要使用 `vm_win10_admin`、`vm_win11_admin`。
- 虚拟机默认不共享本机仓库工作区，跨机产物分发统一走 `http://192.168.86.1`。
- 本文档只描述当前联调环境事实，不承接某一轮测试步骤、测试结果和临时排障记录。
- 如果本机 HTTP 端口、VM 网络或虚拟机名称发生变化，需要先更新本文档，再继续依赖该环境执行联调。

## 6. 推荐使用方式

1. 在本机构建目标产物。
2. 通过 `http://192.168.86.1` 让目标虚拟机下载或覆盖对应文件。
3. 在虚拟机内执行验证、收集日志和观察系统行为。
4. 如果需要跨机协同时序验证，优先固定一台作为 Win10 对照、一台作为 Win11 对照，避免在同一虚拟机中混合多个验证目标。
