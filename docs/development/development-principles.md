# 开发原则

以最小核心闭环为起点构建系统，持续压低复杂度，拒绝兼容包袱，确保结构清晰、边界单一、行为可验证。

## 1. 极简核心原则

内容概述：所有开发先追求最小可成立闭环，只解决当前真实问题，不为未来假设提前堆复杂度。

原则内容：系统先建立一条最短、最直接、可运行、可验证的主路径。抽象、扩展、容错、复用都必须建立在真实需求已经出现的前提上。UI 层遵循 `Element` 法则，能直接使用 `Element Plus` 现成能力时，不额外制造等价复杂度的自定义体系。

该做项：

- 先完成最小核心功能闭环，再考虑补强。
- 新功能默认先做通，再做稳，再判断是否值得抽象。
- 一个能力当前只需要一种实现时，只保留一种实现。
- UI 层优先复用现有成熟组件和能力，不重复造等价复杂度的轮子。

禁止项：

- 为未来可能的扩展、异常、兼容提前设计复杂结构。
- 在没有真实落点时先做多套模式、多层抽象、多组配置。
- 明明只需要一个能力，却提前做成平台化、插件化、可切换化。

## 2. 激进更新原则

内容概述：项目雏形阶段不为旧版本背包袱，旧实现一旦阻碍演进，就直接替换。

原则内容：系统应始终收敛到当前唯一正确的实现，而不是在历史逻辑上持续叠补丁。旧接口、旧结构、旧路径如果已经妨碍推进，应尽快切除，避免早期错误固化为长期成本。

该做项：

- 发现旧实现阻碍演进时，直接替换为新实现。
- 收敛到唯一的新逻辑、新接口、新结构。
- 尽早清理历史包袱，避免错误持续放大。

禁止项：

- 为了旧版本、旧调用、旧行为长期保留双轨逻辑。
- 用补丁叠补丁的方式延续错误结构。
- 把兼容桥、过渡层、中转壳当作长期设计保留。

## 3. 单一真实来源原则

内容概述：一个事实只能有一个最终状态源，一个行为只能有一个最终决策点。

原则内容：状态一旦在多个地方并行维护，就一定会演变为分叉和不一致。上层应通过明确接口读取底层结果，而不是复制状态后各自演化。所有关键状态、配置、生命周期信息，都必须明确唯一归属。

该做项：

- 为每个关键状态明确唯一持有者。
- 上层通过接口读取底层状态，不复制一份再维护。
- 设计阶段主动检查镜像字段、重复缓存、双份状态。

禁止项：

- 同一状态在多个对象、多个层级、多个缓存中并行维护。
- 一个地方改状态，另一个地方补同步，靠约定维持一致。
- 允许状态分叉长期存在，再靠测试或人工经验兜底。

## 4. 边界闸口原则

内容概述：所有输入约束、协议约束、组合约束，都应在边界入口集中处理。

原则内容：边界层负责识别和拦截非法输入，核心层只处理已经成立的数据和流程。约束必须有唯一闸口，不能散落在不同层级反复补判断，否则边界会越来越模糊。

该做项：

- 在入口集中校验输入、协议、组合和范围约束。
- 为每一类约束定义唯一闸口。
- 让核心流程默认运行在合法前提上。

禁止项：

- 在内部多个位置重复兜底同一类条件。
- 把入口治理职责下沉到业务内核。
- 这里放一点校验、那里补一点判断，形成隐式约束网络。

## 5. 结构重构与层次依赖原则

内容概述：模块复杂度一旦明显上升，就要主动重构。重构的目标不是继续包裹复杂度，而是拆开复杂度，恢复清晰职责、清晰层次、清晰边界和清晰依赖。

原则内容：重构不是搬文件，不是加壳，不是套中转层，也不是抽出一个 `service`、`helper` 就结束。重构应把最小组件摊开，下沉到底层，再一层一层向上拼装。默认优先形成多层结构，不允许只拆出一层就停下。依赖组织必须结合语言特性选择：`Go / Python` 优先无环图依赖结构，保证依赖方向单向下沉；`C# / Java / C++` 可以采用受控复杂协作，必要时允许局部环状关系，但不能放任循环依赖掩盖职责混杂。

目录组织再补一条通用固定口径：当一组文件、模块或类型长期共享同一个语义前缀时，默认视为它们已经形成稳定子组件，优先收成 `xxx/` 目录，而不是继续在同一层平铺 `xxx_a`、`xxx_b`、`xxx_c`。这条规则不是只对第一层目录生效，而是在任意第 `N` 层目录都持续生效：只要在 `a/b/` 下面又出现稳定前缀集合，就继续收成 `a/b/xxx/`，而不是长期保留 `a/b/xxx_a`、`a/b/xxx_b`。这本质上就是“按多层结构拆模块”的最直接实践，目录层级本身负责表达组件边界，目录内文件名再表达组件内部角色。例如优先使用 `gateway/storer/link_runtime.go`、`gateway/client/session/opener.go`，而不是长期保留 `gateway/storer_link_runtime.go`、`gateway/client/session_opener.go` 这类前缀模拟分组。只有在该前缀还没有形成稳定边界时，才不强行拆目录。

对 `C / C++ / Qt` 目录组织再补一条固定口径：代码默认按“最小组件目录”组织，而不是按散落单文件组织。一个明确类优先使用“目录 + c/cpp/h/ui”承接；没有明显类归属、但有明确功能边界的代码，按最小组件目录承接，例如 `runtime/`、`media/`、`config/`、`types/`。目标不是把文件四散堆在同一层，而是让目录本身直接表达组件边界。

重构流程：

1. 分析该模块的职责范围和对外调用口，明确它负责什么、被谁依赖、哪里已经混乱。
2. 将功能拆分为最小核心组件，把稳定、通用、低语义的能力下沉到底层。
3. 优先形成清晰多层结构，至少区分“底层最小能力”“中层协作与适配”“上层业务编排与对外接口”。
4. 在上层按真实业务流程逐层拼装这些最小组件，不再让底层混入业务编排职责。
5. 对外重新暴露收束后的调用口，统一入口，收紧边界。
6. 修改其他模块引用，整体切换到新接口，删除旧入口、旧桥接、旧分支。

该做项：

- 当文件接近或超过 `1000` 行、职责混杂、调用绕行、同类逻辑散落、状态与行为出现分叉时，主动触发重构。
- 重构前先识别模块职责、状态归属、边界位置和调用关系。
- 拆分时优先抽出最小核心组件，而不是先抽象大而全的中间层。
- 优先把结构做成多层，而不是抽出一层后就停止。
- 按语言特性组织依赖：`Go / Python` 走无环优先；`C# / Java / C++` 做受控复杂协作。
- 同一语义前缀下的文件、模块、类型一旦开始成组扩张，优先收成目录；这条规则在任意第 `N` 层目录都持续生效：`xxx/detail`、`xxx/detail/subdetail` 都优于长期平铺 `xxx_detail`、`xxx/detail_subdetail`。
- `C / C++ / Qt` 默认按最小组件目录组织代码：优先使用“目录 + c/cpp/h/ui”而不是把同层单文件不断堆平。
- 明确类使用类名目录承接；功能组件使用组件名目录承接。
- 重构完成后统一对外入口，并同步修改所有引用方。

禁止项：

- 把重构做成简单搬文件、改目录、换名字。
- 只抽出一层就停止，导致上下层职责依旧缠绕。
- 保留旧边界、旧入口、旧分支，再在外面包一层新接口。
- 用中转层、兼容层、包装层掩盖原有复杂度。
- 在 `Go / Python` 中做反向包依赖、导入环或隐式双向耦合。
- 在 `C# / Java / C++` 中借“有环协作”之名放任结构失控。
- 已经形成稳定子组件后，继续长期依赖 `xxx_` 前缀平铺文件来模拟目录分组；无论这个平铺发生在第一层还是第 `N` 层目录，均视为同类结构问题。
- 在 `C / C++ / Qt` 中把大量无关实现长期平铺在同一目录，只靠文件名前缀区分组件。

## 6. 删除优先原则

内容概述：删除是控制复杂度的主要手段，不是最后手段。

原则内容：旧分支、坏抽象、冗余封装、失效结构只要已经失去必要性，就应优先删除。每增加一层结构，都必须先回答为什么不能删掉旧结构后直接收束。

该做项：

- 发现坏抽象、旧分支、冗余封装时优先评估删除。
- 新增结构前先判断是否可以通过删除旧结构解决问题。
- 把删除作为日常收束复杂度的正常动作。

禁止项：

- 默认保留旧代码，再在其外面继续叠新代码。
- 不能说明存在必要性的结构继续长期存在。
- 把“以后可能有用”作为保留复杂度的理由。

## 7. 测试覆盖原则

内容概述：测试必须跟随真实风险和真实结构变化，不是只补表层 happy path。

原则内容：每次修改行为边界、状态流转、并发模型、故障恢复路径时，都要同步检查测试覆盖是否仍然成立。测试应优先覆盖真正会出事故的窗口，包括状态竞争、故障切换、边界输入、协议组合、资源释放和时序窗口。

该做项：

- 修改边界、并发、状态、故障恢复时同步补测。
- 优先覆盖状态竞争、故障切换、边界输入、协议组合、资源释放、时序窗口。
- 重构后补新窗口测试，清理已经失效的旧测试。

禁止项：

- 只补表层 happy path，忽略并发、故障、边界场景。
- 代码结构已变化，测试还在验证历史行为。
- 把失效测试、弱相关测试、重复测试长期堆着不清理。

## 8. 文档跟随实现原则

内容概述：文档必须描述当前真实实现，不记录历史幻影，不保留过时分叉。

原则内容：文档的职责是帮助理解现在的系统，而不是保存过去的中间态。结构、边界、支持范围、关键行为一旦变化，文档就必须同步收束，否则文档本身就是错误源。

该原则同样适用于文档本身的重建过程：文档也只保留当前唯一正式口径，不做历史兼容和旧中间态并存。

该做项：

- 文档重点写清职责划分、调用路径、约束条件、支持范围和关键行为。
- 结构、边界、能力范围变化时同步更新文档。
- 把过时文档视为错误信息，及时修正或删除。

禁止项：

- 让文档长期记录历史意图而不是当前事实。
- 代码已经收束，文档仍保留旧结构、旧分支、旧范围描述。
- 堆积无效背景和过时说明，增加理解噪音。

## 开发自检清单

- 这个改动是否先完成了最小核心闭环？
- 是否引入了不必要的扩展点、抽象层或兼容层？
- 这个状态是否只有一个真实来源？
- 这个约束是否已经在唯一边界闸口收住？
- 这个模块是否已经出现职责混杂、层次混乱或依赖反向？
- 当前改动是否可以通过删除旧结构进一步收束？
- 测试是否覆盖了新增窗口，旧测试是否仍然有效？
- 文档描述是否已经与当前实现一致？
