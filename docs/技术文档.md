# 技术文档

补充说明：

- 当前 Windows 驱动链路在 `App-owned media queue` 重建过程中已经积累了较多真实故障案例。
- 与“总体架构/协议意图”不同，这些问题的现象、根因、修复方式和调试手法已经单独整理到 [Windows驱动问题排查笔记](./Windows驱动问题排查笔记.md)。
- 后续遇到 `open control device failed`、`有设备无盘`、`Q2/Q8/Q32 挂死`、`1117/995`、`Ctrl+C 停不掉` 一类问题时，优先先查该笔记，再回看本技术文档。

## 1. 总体架构

系统采用三层结构：

1. `MyApp`
2. `MyCtl`
3. `MyDisk`

逻辑链路如下：

```text
MyApp
  -> 打开 MyCtl 自定义设备接口
  -> DeviceIoControl(MyCtl)
MyCtl
  -> 在内核中定位 GUID_DEVINTERFACE_STORAGEPORT
  -> ZwCreateFile / ZwDeviceIoControlFile
  -> IOCTL_SCSI_MINIPORT
MyDisk
  -> Storport miniport
  -> 对外暴露 SCSI target
系统
  -> disk.sys / classpnp / volmgr / 文件系统
```

在正式建立控制会话之前，还存在一个启动收敛阶段：

```text
MyApp
  -> 单实例锁
  -> 驱动包检查
  -> 设备实例检查
  -> 缺失修复
  -> 重复清理
  -> 连接 MyCtl
```

## 2. 组件职责

## 2.1 MyCtl

### 对外职责

- 自定义 GUID 入口
- App 会话独占
- 用户态协议解析
- 会话关闭触发删盘
- 唯一实例配合识别

### 对内职责

- 枚举 storageport 设备接口
- 打开 `MyDisk` 对应适配器
- 代理 `IOCTL_SCSI_MINIPORT`
- 保持 session 状态

## 2.2 MyDisk

### 对系统职责

- 响应标准 SCSI 指令
- 暴露多 Target 盘
- 负责盘 presence 状态
- 负责总线变化通知

### 对协议职责

- 接收 `MyCtl` 转发的私有命令
- 维护 `WAIT_EVENT` 等待队列
- 维护事务表
- 在收到 App 回包后完成原始 SRB

## 2.3 MyApp

### 用户态职责

- 维护虚拟盘内存后端
- 消费事件
- 回送读数据
- 应答写确认
- 发起建盘和删盘
- 维持心跳和等待深度

## 3. 设备发现

## 3.1 App到MyCtl

`MyApp` 通过 `SetupDiEnumDeviceInterfaces` 枚举 `MyCtl` 自定义 GUID。  
不再直接枚举 `GUID_DEVINTERFACE_STORAGEPORT`。

在进入接口枚举前，`MyApp` 必须先完成本地唯一性收敛：

- 通过命名互斥体或等效锁保证单进程
- 通过 SetupAPI/ConfigMgr 枚举 `Root\\MyKWDF3`
- 通过 SetupAPI/ConfigMgr 枚举 `Root\\MyDisk2`

## 3.1.1 驱动包安装

建议 App 侧使用标准驱动安装函数完成驱动包预装和安装。

推荐顺序：

1. 检查 INF 是否已在 Driver Store 中可用
2. 若不可用，调用 `DiInstallDriver` 预装驱动包
3. 后续若创建设备实例，PnP 绑定已在 Driver Store 中的匹配包

相关官方文档：

- `DiInstallDriver`  
  https://learn.microsoft.com/en-us/windows/win32/api/newdev/nf-newdev-diinstalldriverw

## 3.1.2 根设备实例创建

当前两个驱动都是 Root 枚举模型，因此缺失设备实例时应由 App 显式创建 devnode。

建议机制：

1. `SetupDiCreateDeviceInfoList`
2. `SetupDiCreateDeviceInfo`
3. `SetupDiSetDeviceRegistryProperty` 写入硬件 ID
4. `SetupDiCallClassInstaller(DIF_REGISTERDEVICE)`

相关官方文档：

- `SetupDiCreateDeviceInfo`  
  https://learn.microsoft.com/en-us/windows/win32/api/setupapi/nf-setupapi-setupdicreatedeviceinfoa
- `DIF_REGISTERDEVICE`  
  https://learn.microsoft.com/en-us/previous-versions/windows/drivers/install/dif-registerdevice

## 3.1.3 重复设备清理

如果枚举结果多于一个，则按固定策略保留一个，删除其余实例。

建议删除接口：

- `SetupDiRemoveDevice`
- 或 `DiUninstallDevice`

本项目优先按“删除多余 devnode，不移除驱动包”处理。

相关官方文档：

- `SetupDiRemoveDevice`  
  https://learn.microsoft.com/en-us/windows/win32/api/setupapi/nf-setupapi-setupdiremovedevice

## 3.2 MyCtl到MyDisk

`MyCtl` 在内核中枚举 `GUID_DEVINTERFACE_STORAGEPORT`，再通过以下方式过滤出 `MyDisk`：

- 设备实例 ID
- Service 名称
- FriendlyName
- `QUERY_INFO` 私有命令验证签名

推荐最终以 `QUERY_INFO` 返回固定签名和协议版本作为确认手段。

## 4. 磁盘模型

## 4.1 多盘方式

使用多 Target 模型：

- `Path = 0`
- `Target = 0..N-1`
- `Lun = 0`

不使用单 Target 多 LUN 模型。

## 4.2 Presence模型

每个磁盘对象必须至少有以下状态：

- `Configured`
- `Present`
- `Removing`
- `Offline`

初始状态：

- `Configured = FALSE`
- `Present = FALSE`

额外约束：

- 整个系统只能存在一个 `MyDisk` 适配器设备实例
- 多块盘都挂在这一个适配器下，以多 Target 形式暴露

## 4.3 容量与块大小

每块盘应至少记录：

- `SectorSize`
- `SectorCount`
- `DiskSize`
- `TargetId`
- `Generation`

`READ_CAPACITY` 和 `READ_CAPACITY16` 必须都正确实现。  
超过 2TB 的介质必须保证 `READ_CAPACITY16` 返回真实容量，`READ_CAPACITY` 返回 `0xFFFFFFFF` 封顶语义。

## 5. 协议设计

协议分为两层：

- `MyApp <-> MyCtl` 用户态控制协议
- `MyCtl <-> MyDisk` 私有 miniport 协议

用户态协议可以与 miniport 私有协议保持同构，但建议保留独立版本号。

## 5.1 命令分类

### APP_CMD

同步或短时命令：

- `QUERY_INFO`
- `CREATE_DISK`
- `REMOVE_DISK`
- `REMOVE_ALL_DISKS`
- `READ_REPLY`
- `WRITE_ACK`
- `FETCH_WRITE_DATA`
- `HEARTBEAT`
- `LIST_DISKS`

### WAIT_EVENT

长等待命令，用于把 `MyDisk` 里的事件送到 `MyApp`。

## 5.2 通用头

建议所有协议包统一使用如下头部：

```c
typedef struct _MYVDSK_HEADER {
    UINT32 Version;
    UINT32 Command;
    UINT64 SessionId;
    UINT64 TxId;
    UINT32 TargetId;
    UINT32 Flags;
    UINT32 PayloadLength;
    UINT32 Status;
} MYVDSK_HEADER;
```

## 5.3 事件类型

建议事件类型如下：

- `EVENT_READ_REQ`
- `EVENT_WRITE_REQ_INLINE`
- `EVENT_WRITE_REQ_HDR`
- `EVENT_DISK_ADDED`
- `EVENT_DISK_REMOVED`
- `EVENT_SHUTDOWN`

## 5.4 事务ID

读写都必须有 `TxId`。

`TxId` 用途：

- 唯一标识一个系统原始 I/O
- 关联 pending SRB
- 匹配 `READ_REPLY`
- 匹配 `WRITE_ACK`
- 支持日志追踪和超时处理

## 5.5 大数据写入

对于写请求，推荐采用以下规则：

- 小写入：事件中内联正文
- 大写入：事件中只发头部，App 再通过 `FETCH_WRITE_DATA` 分段拉取

这样可以避免 64 个 `WAIT_EVENT` 都携带大缓冲区导致的内存压力。

## 6. 状态机

## 6.1 会话状态机

```text
NoSession
  -> AppOpen
SessionActive
  -> WaiterReady
Running
  -> HeartbeatLost / FileCleanup / AppClose
RemovingAllDisks
  -> NoSession
```

## 6.1.1 启动收敛状态机

```text
AppStart
  -> AcquireSingleInstance
  -> EnsureDriverPackages
  -> EnsureMyCtlDevnode
  -> EnsureMyDiskDevnode
  -> RemoveDuplicateMyCtl
  -> RemoveDuplicateMyDisk
  -> OpenMyCtl
  -> SessionActive
```

## 6.2 事务状态机

```text
Allocated
  -> EventQueued
  -> EventDelivered
  -> ReplyReceived
  -> SrbCompleted
```

异常路径：

- `Timeout`
- `DiskRemoved`
- `SessionLost`
- `Cancelled`

这些异常路径都必须进入失败完成。

## 7. 并发与锁

### MyCtl

建议至少维护以下同步对象：

- 全局会话锁
- 下层 storageport 句柄锁
- 挂起 `WAIT_EVENT` 请求列表锁
- 驱动自修复期间的全局安装锁

### MyDisk

建议至少维护以下同步对象：

- 盘数组锁
- 事务表锁
- 等待事件队列锁
- 每盘状态锁

## 8. 热插拔策略

删除盘时的动作顺序必须固定：

1. 标记目标盘 `Present = FALSE`
2. 从事务表中摘除相关事务
3. 失败完成相关 pending SRB
4. 丢弃或取消等待中的相关事件
5. 通知 Storport 总线变化

该动作等效于“设备突然拔除”，不要求文件系统优雅卸载。

## 9. 超时策略

建议初始参数：

- 心跳周期：1000 ms
- 心跳超时：3000-5000 ms
- `WAIT_EVENT` 最低深度：8
- 正常预投深度：64
- 读写事务超时：10000-30000 ms

## 9.1 单实例策略

`MyApp` 建议在进程启动最前面使用命名互斥体保证单实例。

建议名：

- `Global\\MyApp5.SingleInstance`

相关官方文档：

- `CreateMutex`  
  https://learn.microsoft.com/en-us/windows/win32/api/synchapi/nf-synchapi-createmutexw

说明：

- `CreateMutex` 可用于单实例控制
- 如果发现 `ERROR_ALREADY_EXISTS`，说明已有实例存在

## 10. 错误处理

### 会话丢失

- 删除全部盘
- 失败完成全部事务
- 清空队列

### 启动收敛失败

- 不进入工作态
- 不允许建盘
- 记录清晰错误并退出

### App回包错误

- 对应事务失败完成
- 记录日志

### 目标盘不存在

- 直接返回设备不存在类错误

### 协议版本不匹配

- 拒绝会话
- 返回协议信息
