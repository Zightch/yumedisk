# 工作流

本文档定义当前仓库在实现阶段的统一执行工作流，约束 `docs/tmp/todo.md`、`docs/progress/`、正式文档和 Git 提交之间的配合方式。

更新时间：2026-04-20

## 1. 文档分工

- `docs/workflow.md` 负责定义统一执行流程、`todo` 轮换规则、归档时机、`.gitignore` 检查和提交约定。
- `docs/development-principles.md` 负责定义仓库级统一开发原则，不承接当前轮临时推进过程。
- `docs/progress/README.md` 负责说明进度归档目录的使用方式和索引。
- `docs/progress/*.md` 负责按日期承接跨轮历史进度、停止线、已完成项和已确认边界。
- `docs/tmp/todo.md` 只负责当前轮未完成执行项，必须明确当前总目标、实现该目标的子步骤、当前轮边界，以及当前唯一下一步，不记录已完成内容。
- `docs/tmp/` 可以作为临时文档草稿箱使用，但除 `todo.md` 外，其余草稿默认不纳入版本管理。

## 2. Todo 轮换约束

- 当前轮执行只看 `docs/tmp/todo.md`。
- `docs/tmp/todo.md` 只保留当前总目标、该目标下的子步骤、当前轮边界和当前唯一下一步。
- `todo` 中不写已完成内容；已完成内容一律即时归档到 `docs/progress/`，不回填到当前轮 `todo`。
- 当前唯一下一步必须保持单一明确，不并列堆叠多个动作。
- 当前总目标完成后，先完成归档和提交，再立即清空当前 `todo`，等待下一轮重新写入。

## 3. 进度归档规则

- 进度归档按日期写入 `docs/progress/YYYY-MM-DD.md`。
- 同一天完成的多个子步骤，继续追加到同一天文件，不额外新建“当前进度”总表。
- 历史修改如果没有明确完成时间，可以参考 `git log` 选择最接近的日期近似补档，并在归档文件中注明“近似归档”。
- 进度归档只记录已完成事实、验收结果和已收束边界，不保留当前待办。

## 4. 固定执行步骤

1. 完成即归档：任一子任务完成后，立刻把完成项、验收结果和已经稳定的边界同步到对应日期的 `docs/progress/*.md`；如果形成了长期有效规则，再同步到正式文档。
2. 检查忽略并提交：每一子步骤完成后，检查是否产生了新的临时文件、构建产物或本地环境文件需要补进 `.gitignore`；补齐后，立即提交当前稳定工作内容。
3. 提炼当前目标：当前 `todo` 只保留仍待推进的最小总目标、该目标下的子步骤、边界和下一步，不复制已完成项，不搬运上一轮讨论过程。
4. 收紧当前下一步：如果当前唯一下一步写成了并列多项，改写成一个清晰动作后再执行。
5. 重写当前 todo：直接重写 `docs/tmp/todo.md`，只保留当前总目标、实现该目标的子步骤、当前轮边界、当前唯一下一步，以及对 `docs/progress/README.md` 的引用。
6. 按当前轮持续更新：`todo` 只增删未完成工作；如果当前总目标已经完成，则先完成归档和提交，再立即清空当前 `todo`。
7. 单步停止：每完成一个子步骤并完成归档、提交、todo 更新后，立刻停下，不继续执行下一个子步骤，等待用户确认后再推进。
