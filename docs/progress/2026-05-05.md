  - 新数据面入口已切到 `PostReadSlot / PostWriteSlot / ReadAck / WriteAckBatch / CancelSlot` 语义，并在 KMDF 侧把 slot / ack 转成 `SubmitSlot / ReadAck / WriteAckBatch / CancelSlot` 小描述符转发给 SCSI。

- `windows/tests/RWTestApp/main.cpp` 与 `windows/tests/MyApp5/main.cpp` 已改成新的 `DeviceIoControl(input=message, output=buffer)` 收包方式，匹配新的 KMDF 边界解释。
- 完成“Rework KMDF session handling around file-bound lifecycle and locked timeout state”子步骤。
- `windows/YumeDiskKMDF/YumeDiskKMDF` 的 session 生命周期已从 device-global 下沉到 file-bound：
  - `CTRL_FILE_CONTEXT` 现在持有 `SessionLock / WatchdogTimer / MiniportHandle / SessionId / LastHeartbeatTick / State`。
  - `CTRL_DEVICE_CONTEXT` 只保留独占打开控制，不再缓存数据面默认 miniport handle。
- `ControlSessionTryOpen` 现在在 App 打开 file object 时立即建立 session、打开 miniport handle、启动 watchdog；`ControlSessionCleanup` 在 file cleanup 时固定执行 `RemoveAllDisks | SessionClose -> close miniport handle -> release open slot`。
- heartbeat 机制已收敛为仅在 KMDF 内维护：
  - `Heartbeat` IOCTL 只更新 KMDF session 的 `LastHeartbeatTick`。
  - `YumeDiskSCSI` 不再承担 heartbeat 机制。
- watchdog 超时后，KMDF 会把该 file session 切到 locked 状态，发送 `RemoveAllDisks | SessionClose` 清盘并取消后端，再关闭自己持有的 miniport handle；在 file object 真正关闭前，该 session 的后续 IOCTL 直接失败。
- `transport` 层已改成基于 file session 工作：
  - `ControlOpenMiniportHandle(...)` 显式打开 file-bound miniport handle。
  - `ControlProxyCommand(...)`、`ControlCloseMiniportHandle(...)`、`ControlSendSessionCleanup(...)` 都改为围绕 `CTRL_FILE_CONTEXT` 工作。
- `control/ioctl.c` 已改为从 `WdfRequestGetFileObject(Request)` 定位 session；除 heartbeat 外，其余命令都先获取 file session 再透传，避免 handle/state 与 watchdog cleanup 竞态。
- `windows/tests/RWTestApp/main.cpp` 已增加自动 heartbeat 后台线程；`windows/tests/MyApp5/main.cpp` 已增加手工 `heartbeat` 命令，便于 session 生命周期调试。
- 完成“Rewrite SCSI queue processing to drain while work is available and split lock scope”子步骤。
- `windows/YumeDiskSCSI/YumeDiskSCSI/core/defs.h` 已把 miniport 队列状态从单一控制锁扩成三段：
  - `SessionLock` 只管 session claim / close。
  - `ReadQueueLock` 只管 posted read slots 与 pending reads。
  - `WriteQueueLock` 只管 posted write slots、pending writes 与 ACK 落账。
  - `NextEventId` 改成 interlocked 自增，不再依赖单大锁。
- `windows/YumeDiskSCSI/YumeDiskSCSI/queue/queue.c` 已从空壳改成真实数据面队列：
  - `SubmitSlot` 现在会把 App 预投 read/write slot 变成长期 pending miniport control SRB。
  - READ SRB 会排入 pending read 表，命中同 target 的 posted read slot 后立刻回填 `YUMEDISK_READ_SLOT_EVENT` 并完成 slot。
  - WRITE SRB 会排入 pending write 表，消费 posted write slot 时在锁外复制片段数据并回填 `YUMEDISK_WRITE_SLOT_HEADER`。
  - `ReadAck` 会按 `eventId` 找回原始 READ SRB，把 App buffer 拷回系统 buffer 后完成请求。
  - `WriteAckBatch` 会按 bitmap 落账 `eventId + seq`，全部确认后完成原始 WRITE SRB；单个 fragment 失败只打死对应大 WRITE，不影响后续 WRITE。
  - `CancelSlot`、`RemoveDisk`、`RemoveAllDisks`、`SessionClose` 都会把对应 target 或整 session 的 pending slots / pending SRBs 清掉，而不是把 stale 项留在队列里。
- 请求推进已经改成 `drain while available`：
  - READ/WRITE/slot submit/ACK 进入后，都会持续循环消费当前已经可匹配的 slot 与 pending request，而不是只推进一个就等下一次异步唤醒。
  - 大数据 copy、App slot completion、系统 SRB completion 都被移到锁外，避免高队列深度下把 spin lock 变成 CPU 热点。
- `windows/YumeDiskSCSI/YumeDiskSCSI/scsi/scsi.c` 已不再直接拒绝 READ/WRITE：
  - 现在会解析 READ/WRITE(6/10/12/16) CDB 的 LBA 与 block count。
  - 合法 READ/WRITE 会进入新的 pending queue 路径；非法长度或非法 CDB 仍在边界返回错误。
- `windows/YumeDiskSCSI/YumeDiskSCSI/control/control.c` 的新协议命令已经真正接到队列层：
  - `SubmitSlot` 可挂起并在稍后由真实 I/O 完成。
  - `ReadAck` / `WriteAckBatch` / `CancelSlot` 不再只是 `STATUS_NOT_SUPPORTED` 空壳。
  - `PostWriteSlot` 自带的 ACK trailer 现在会在 SCSI 侧先消费，再续租 write slot credit。
- `windows/shared/yumedisk_proto.h` 已把 `YUMEDISK_MINIPORT_TIMEOUT_SEC` 从 `30s` 提高到 `3600s`，避免长期 pending slot 在当前实现下天然踩到 miniport control timeout。

## 已收束边界

- 当前仓库已经没有旧 `WaitEvent` 数据面主路径。
- 当前骨架阶段只保留控制面闭环，读写压测能力暂时故意断开。
- 下一步必须从新协议入口开始重建，不能回退到旧 event inline data 模型。
- 当前新协议入口、KMDF session/watchdog、SCSI real queue 都已经落下；剩余缺口已经收缩到 App backend worker 与 benchmark 闭环，不再是驱动侧协议骨架问题。
- heartbeat/watchdog 的唯一真实来源已经收敛到 KMDF file session；SCSI 不再承担 App 存活探测职责。
- miniport handle 生命周期已经与 App file object 绑定，不再允许回退到 device-global lazy cached handle 模型。
- session 一旦 watchdog 超时进入 locked 状态，在 file object 关闭前不允许通过新的 heartbeat 或新的 slot 请求复活。
- 当前 SCSI 已经具备真实 pending queue 和 while-drain 行为，下一步不应再回退到“每次只处理一个、靠下一次唤醒继续推进”的单步模型。
- 当前实现仍然使用长期 pending `IOCTL_SCSI_MINIPORT` slot SRB，只是先把 timeout 拉长到 `3600s`；真正摆脱这类长期 pending transport 风险仍留给后续更深一层的收束。
- Benchmark 最小闭环还差 App backend 并发调度与压测路径重连；当前停止线是“驱动侧队列已通，用户态 worker 仍未重建”。
