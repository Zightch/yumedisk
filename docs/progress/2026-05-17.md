# 2026-05-17

## 已完成

### 1. storer 自带网关单盘最小闭环已打通

- `server/internal/storer` 当前已完成自带 `embedded gateway` 的单盘最小闭环。
- `AuthStart / AuthFinish / SessionOpen / ReadAt / WriteAt / Ping / Close` 已贯通。
- `storer` 本地 `raw` 文件后端已可承接真实块读写。
- 当前 `SessionOpen` 已固定为单盘独占策略：
  - 多个 client 可以各自认证成功；
  - 但同一时刻只允许一个活跃打开会话；
  - 后续 client 在 `SessionOpen` 阶段拿到 `disk-busy`。

### 2. 会话语义已收口为 connection-scoped 认证资格 + session-scoped 数据面

- 认证阶段定位符当前是 `challenge_token`，而不是对 client 额外暴露独立 `auth_id`。
- 认证成功后，gateway 只在当前连接上记录该 `disk_id` 的认证资格。
- 真正的数据面定位符仍然只有 `session_id`。
- 已确认当前边界成立：
  - 认证成功不等于拿到会话；
  - `SessionOpen` 才是真实会话创建入口；
  - `SessionOpen` busy 失败后，只要连接未断开，允许在同一连接上再次重试 `open`。

### 3. 新增协议联调工具 `network-auth-open`

- 新增 `windows/rust-cli/src/bin/network-auth-open.rs`。
- 该工具只复用 `rust-cli` 现有 `network` 模块，不经过 `BackendRust / AppKernel / KMDF`。
- 当前已重建为常驻交互联调程序，命令为：
  - `auth`
  - `open`
  - `close`
  - `state`
  - `exit`

用途：

- 保持真实 `GatewayConnection`
- 分开观察认证资格和真实会话的边界
- 复现 `disk-busy`
- 验证“同一连接上重试 `SessionOpen`”的协议语义

### 4. 已完成真实联调验证

通过 `network-auth-open` 与真实 `storer` 常驻进程已验证下面这条链：

1. A 连接 `auth`
2. A `open` 成功
3. B 连接 `auth`
4. B 第一次 `open` 返回 `disk-busy`
5. A `close`
6. B 不重连、不重新认证，在同一连接上再次 `open`
7. B 第二次 `open` 成功，拿到新的 `session_id`

这证明当前“认证资格绑定连接、会话独立创建”的模型已经能稳定工作。

### 11. 已完成 B3 gateway session 映射层

- `server/internal/gateway` 已新增独立 `sessionRegistry`，专门维护：
  - `gateway_session_id`
  - `client_connection_id`
  - `route_connection_id`
  - `upstream_session_id`
- `SessionOpen` 当前已不再把底层 `session.Service` 返回的 `session_id` 直接回给 client：
  - `gateway` 会先保存 `upstream_session_id`
  - 再分配自己的 `gateway_session_id`
  - client 后续只看到 `gateway_session_id`
- `Ping / ReadAt / WriteAt / Close` 当前都已改为：
  - 先查 `gateway_session_id -> upstream_session_id`
  - 再对底层数据面使用真实 upstream session id
- 失效清理当前已收口为：
  - `Close(session_id)` 时同时删除 gateway 映射并关闭 upstream session
  - client 连接关闭时，同时删除该连接名下全部 gateway session 映射，并逐个关闭 upstream session
  - 若底层已返回 `session unavailable`，gateway 会同步删除本地映射
- 这意味着当前 `client-and-gateway` 边界已经不再暴露底层 session 命名空间，为后续真正的 `gateway <-> storer` 转发链保留了稳定外部接口。
