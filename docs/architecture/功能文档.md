# 功能文档

## 1. 功能范围

YumeDisk 提供由 App 控制的虚拟数据盘能力，包含驱动与设备自修复、会话管理、盘管理、app-owned slot 数据面、读写 ACK、异常退出清盘和压测验证能力。

当前功能文档以最终多盘效果为准：`YumeDiskKMDF`、`YumeDiskSCSI`、`TestApp` 共同落到 app-owned media queue。早期 `APP_CMD + WAIT_EVENT` 事件模型只作为历史背景，不再作为功能目标。

## 2. 组件功能

## 2.1 YumeDiskKMDF

`YumeDiskKMDF` 是 App 可见的控制入口和 session-aware transport 层。

### 必选功能

- 暴露自定义设备接口 GUID，供 App 枚举和打开。
- 仅允许一个 active App 会话。
- 感知 `CreateFile`、`Cleanup`、`Close`。
- 在内核中定位并打开目标 `YumeDiskSCSI` miniport。
- 使用 `QueryKmdfInfo` / `QueryScsiInfo` 确认组件版本和目标 miniport 身份；其中兼容性只看版本，签名只用于识别目标 miniport。
- 维护 file-bound session、session id、watchdog 和 locked/closed 状态。
- App file cleanup 或 watchdog 超时时，按 `RemoveAllDisks | SessionClose -> drain in-flight slot IO -> close miniport handle` 清理。
- 使用 direct I/O 取得 App buffer 的 system VA。
- 代理短控制命令：`QueryScsiInfo`、`CreateDisk`、`RemoveDisk`、`RemoveAllDisks`、`QueryDebugState`；本地回答 `QueryKmdfInfo`。
- 代理数据面命令：`POST_READ_SLOT`、`POST_WRITE_SLOT`、`READ_ACK`、`WRITE_ACK_BATCH`。
- 对 `POST_READ_SLOT` / `POST_WRITE_SLOT` 使用异步 slot transport，只有启动失败才同步完成错误。

### 不做的事情

- 不保存或拥有虚拟盘介质。
- 不做普通文件系统 I/O。
- 不做 LBA overlap 检测。
- 不维护自己的数据面调度队列。
- 不重新包装旧 `WAIT_EVENT` inline payload 兼容层。

## 2.2 YumeDiskSCSI

`YumeDiskSCSI` 是 Storport miniport，负责向 Windows 暴露磁盘并完成原始 SCSI SRB。

### 必选功能

- 暴露多个 Target，每个 Target 只有 LUN 0。
- 默认无 App 会话时不暴露数据盘。
- 支持建盘、删盘和清空全部盘。
- 支持按盘声明系统只读属性，并由 `SCSI` 统一宣告写保护、统一拒绝系统写请求。
- 支持 `INQUIRY`、`REPORT_LUNS`、`READ_CAPACITY`、`READ_CAPACITY16`。
- 支持 `READ/WRITE/READ16/WRITE16`。
- 支持私有 miniport 控制协议入口。
- 每个 target 独立维护 read queue 和 write queue。
- 对 READ SRB 分配 `eventId`，挂入 `PendingReadsByEventId`，等待 read slot 与 `READ_ACK`。
- 对 WRITE SRB 分配 `eventId`，按 write slot 容量拆成 `seq/totalSeq` fragment，使用 bitmap 处理 `WRITE_ACK_BATCH`。
- 处理 stale/cancelled ACK 时返回逐项失败结果，不因单项失败打断整条 ACK 控制命令。
- 固定 `WRITE_ACK_BATCH` 顺序：先完成 ACK 控制请求回 App，再完成因此满足条件的系统 WRITE SRB。
- session close、reset、remove、cancel 路径必须取消 pending slot 并失败完成 pending SRB。

### 运行约束

- READ/WRITE 单请求失败只影响对应请求，不把磁盘判死。
- 不维护 heartbeat；App 存活只归 `YumeDiskKMDF` 管。
- 不保存已经完成给 App 的 App buffer VA。
- 不让多盘共享 adapter 级数据面队列和全局读写锁。

## 2.3 TestApp

`TestApp` 是 App 侧实现和验证入口，最终必须按磁盘隔离数据面调度。

### 必选功能

- 单实例运行。
- 启动时扫描驱动包和设备实例。
- 自动安装缺失驱动包。
- 自动创建缺失设备实例。
- 自动清理多余设备实例。
- 枚举并打开 `YumeDiskKMDF` 自定义设备接口。
- 建立独占控制会话并周期性 heartbeat。
- 请求创建和删除磁盘。
- 为每个磁盘维护独立 medium。
- 为每个磁盘维护独立 read/write workers、独立 write ACK flush worker 和 queue depth。
- 持续预投 read slots 和 write slots。
- 接收 read slot 后读取 medium，并发送 `READ_ACK`。
- 接收 write slot 后写入 medium，并通过独立 `WRITE_ACK_BATCH` flush。
- 打印 `WRITE_ACK_BATCH` 逐项失败，便于区分 stale/cancelled ACK。
- 提供 `Q1T1 / Q8 / Q32` benchmark 验证入口。

## 2.4 BackendRust 与 tauri-client

`BackendRust` 与 `tauri-client` 是当前正式桌面客户端实现，负责把最小可用 UI 与 `AppKernel` / `BackendRust` 宿主能力接起来。

### 必选功能

- 程序启动后直接进入主页，不再单独保留初始化页路由。
- 从用户目录 `.yumedisk` 读取客户端配置，并恢复当前进程内 `DiskRuntime` 列表。
- 为每个 `DiskRuntime` 维护唯一 `media` 持有语义：
  - 内存盘 `media` 在当前 App 常驻期内常驻；
  - 文件盘 `media` 在拔出或转为无效时置空；
  - 无效盘不持有 `media`。
- 提供 `restore_client_state` 与 `open_session` 两个独立宿主入口，不再允许重新合并成单一初始化命令。
- 在主页启动时按唯一顺序执行“恢复配置 -> 查询 runtime 磁盘 -> 打开会话 -> 启动重扫 -> 退出启动覆盖 -> 自动挂载”。
- 提供顶部会话状态入口，统一承接初始化中、会话正常、会话失败三态展示。
- 提供会话失败后的唯一重试入口；重试只覆盖“打开会话 -> 启动重扫 -> 自动挂载”链路，不重复恢复配置。
- 提供磁盘创建、挂载、拔出、编辑、删除、重扫的最小闭环。
- 关闭主窗口时转入托盘常驻；托盘重新显示主窗口时不重新执行启动自动挂载。

### 不做的事情

- 不把“正在初始化 / 会话失败”写回 Backend 真实磁盘状态。
- 不在前端复制 `AppKernel` worker、队列、暂存层等数据面逻辑。
- 不建立运行期 `session broken` 自动自愈和后台自动重连体系。
- 不在托盘显示主窗口时重新跑整段主页启动链。

## 3. 功能清单

## 3.1 驱动与设备自修复

App 每次启动都必须做如下动作：

1. 检查 `YumeDiskKMDF` 驱动包是否已安装。
2. 检查 `YumeDiskSCSI` 驱动包是否已安装。
3. 如缺失，则安装驱动包。
4. 检查 `YumeDiskKMDF` 设备实例数量。
5. 检查 `YumeDiskSCSI` 设备实例数量。
6. 如数量为 0，则创建设备实例。
7. 如数量大于 1，则保留一个规范实例，删除多余实例。

规范实例选择规则建议：

- 优先保留已启用且无 problem code 的实例。
- 若存在多个健康实例，保留设备实例 ID 最小的一个。
- 其余实例全部删除。

## 3.2 盘管理

### 创建磁盘

输入：

- 目标 `TargetId`
- 容量
- 扇区大小
- `ReadOnly`

约束：

- 必须已有有效 App session。
- 一个 `TargetId` 同时只能对应一块盘。
- 建盘成功后由 `YumeDiskSCSI` 通知总线变化。
- `ReadOnly != 0` 时，对系统暴露为只读盘；只读写保护由 `SCSI` 单点生效，不在 `AppKernel` 和业务层并行决策。

### 删除磁盘

输入：

- 指定 `TargetId`，或删除全部。

行为：

- 立即将目标盘标记为 not present。
- 取消该盘 pending slots。
- 失败完成该盘 pending READ/WRITE SRB。
- 通知系统总线变化。

删盘按“物理热拔出”的激进策略处理，不保证文件系统优雅卸载。

## 3.3 会话管理

### 会话建立

- App 打开 `YumeDiskKMDF`。
- `YumeDiskKMDF` 建立唯一 session。
- `YumeDiskKMDF` 打开并验证 `YumeDiskSCSI` miniport handle。

### 会话保活

- App 周期性发送 heartbeat。
- `YumeDiskKMDF` 检查 heartbeat 超时。
- heartbeat 不转发给 `YumeDiskSCSI`。

### 会话销毁

以下任一情况都视为会话结束：

- App 主动关闭句柄。
- App 进程崩溃。
- heartbeat 超时。
- KMDF file cleanup。

结果：

- 删除全部磁盘。
- 取消 pending App slots。
- 失败完成 pending system SRB。
- 锁定该 session，直到 file object 关闭。

补充约束：

- 系统侧取消逻辑只追到 `SCSI`。
- `App` 不维护额外的全链路取消协议。
- App 的 late `READ_ACK` / `WRITE_ACK_BATCH` 到达后，由 `SCSI` 判断对应系统请求是否已经取消或不存在。

## 3.4 系统读盘流程

1. 系统向 `YumeDiskSCSI` 发出 READ。
2. `YumeDiskSCSI` 为该请求分配 `eventId`。
3. `YumeDiskSCSI` 消费该 target 的 posted read slot。
4. `YumeDiskKMDF` 完成 App 的 `POST_READ_SLOT`。
5. App 从自己的 medium 读取数据。
6. App 通过 `READ_ACK(eventId, data)` 回包。
7. `YumeDiskSCSI` 复制数据并完成原始 READ SRB。

## 3.5 系统写盘流程

1. 系统向 `YumeDiskSCSI` 发出 WRITE。
2. `YumeDiskSCSI` 为原始 WRITE 分配 `eventId`。
3. `YumeDiskSCSI` 按 write slot capacity 拆出 `seq/totalSeq`。
4. `YumeDiskSCSI` 消费该 target 的 posted write slots。
5. `YumeDiskKMDF` 完成 App 的 `POST_WRITE_SLOT`。
6. App 将 fragment 写入自己的 medium。
7. App 通过独立 `WRITE_ACK_BATCH` 确认一个或多个 `eventId + seq range`。
8. `YumeDiskSCSI` 用 bitmap 落账，全部 fragment 确认后完成原始 WRITE SRB。

## 3.6 ACK 语义

- `POST_WRITE_SLOT` 输入 payload 固定为空，不再携带 ACK trailer。
- `WRITE_ACK_BATCH` 是唯一写确认入口。
- 系统侧取消只追到 `SCSI`；`App` 只需要继续按正常路径回 ACK。
- `SCSI` 在 ACK 到达时再判断对应系统请求是否仍存在。
- 单个 ACK range 找不到 `eventId` 时，返回该 range 的失败结果，例如 `STATUS_NOT_FOUND`。
- 单个 range 失败不应导致整条 `WRITE_ACK_BATCH` 命令失败。
- 重复 ACK、越界 ACK、未投递 seq ACK 必须被显式识别，便于定位 App 侧错误或取消竞态。

## 3.7 桌面客户端启动与恢复

当前正式桌面客户端的主页启动顺序固定如下：

1. `restore_client_state`
2. `query_home_disk_list`
3. `open_session`
4. `rescan_runtime_disks`
5. 退出主页启动覆盖
6. 首次自动挂载

固定约束：

- `restore_client_state` 只负责恢复配置和宿主运行态，不负责打开 `AppKernel` 会话。
- `open_session` 只负责打开会话并返回会话快照，不负责恢复配置。
- `rescan_runtime_disks` 只重扫当前进程内运行时磁盘列表，不重读配置文件。
- `rescan_runtime_disks` 不枚举 Windows 系统设备路径，不等待 `PhysicalDrive` 出现。
- 首次自动挂载严格依赖“启动重扫后的快照”，不允许在重扫前提前触发。
- 托盘重新显示主窗口不重复执行首次自动挂载。

主页启动阶段的磁盘展示固定采用前端展示覆盖：

- 会话尚未完成启动重扫前，主页统一把磁盘显示为 `invalid`。
- 默认无效原因显示为 `正在初始化`。
- 会话打开失败后，无效原因统一切为 `会话失败`。
- 启动重扫成功后，主页才退出该覆盖并恢复 Backend 真实磁盘状态。

这层覆盖只允许存在于主页单一展示映射口，不允许散落到：

- Backend 真实状态模型；
- 磁盘卡片组件内部；
- API DTO 补字段；
- 各个操作按钮局部补判断。

## 4. 非功能要求

## 4.1 稳定性

- App 不在线时绝不保留盘。
- App 启动时必须把驱动与设备状态收敛到唯一有效形态。
- session close 必须完成或取消所有 pending App slot 和 system SRB。
- `Q2/Q8/Q32` 不应出现 slot 丢失后系统 SRB 永久 pending。

## 4.2 可调试性

- 三个组件都要带清晰日志。
- 每个读写请求都可打印 `eventId`。
- 写 fragment 必须可打印 `eventId + seq/totalSeq`。
- 每个盘都要有 `TargetId`。
- `WRITE_ACK_BATCH` 逐项失败要能在 App 侧观察。

## 4.3 并发性

- 支持多盘。
- 支持并发系统 I/O。
- 支持 per-target SCSI 队列。
- 支持 per-disk App worker 组。
- 只允许单 App 控制会话。

## 4.4 唯一性

- App 进程唯一。
- `YumeDiskKMDF` 设备实例唯一。
- `YumeDiskSCSI` 设备实例唯一。
- `YumeDiskKMDF` active session 唯一。
