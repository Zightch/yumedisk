# 技术文档

补充说明：

- 当前 Windows 驱动链路在 `App-owned media queue` 重建过程中已经积累了较多真实故障案例。
- 与“总体架构/协议意图”不同，这些问题的现象、根因、修复方式和调试手法已经单独整理到 [Windows驱动问题排查笔记](../troubleshooting/Windows驱动问题排查笔记.md)。
- 后续遇到 `open control device failed`、`有设备无盘`、`Q2/Q8/Q32 挂死`、`1117/995`、`Ctrl+C 停不掉` 一类问题时，优先先查该笔记，再回看本技术文档与本文第 5 节协议设计。

## 1. 总体架构

当前系统采用三层结构：

1. `TestApp`
2. `YumeDiskKMDF`
3. `YumeDiskSCSI`

逻辑链路如下：

```text
TestApp
  -> CreateFile(YumeDiskKMDF custom interface)
  -> DeviceIoControl(YumeDiskKMDF)
YumeDiskKMDF
  -> session/watchdog/direct-I/O transport
  -> locate YumeDiskSCSI storageport interface
  -> IOCTL_SCSI_MINIPORT / async slot IRP
YumeDiskSCSI
  -> Storport miniport
  -> expose SCSI targets
Windows
  -> disk.sys / classpnp / volmgr / file systems
```

在正式建立控制会话前，App 先完成启动收敛：

```text
AppStart
  -> AcquireSingleInstance
  -> EnsureDriverPackages
  -> EnsureKMDFDevnode
  -> EnsureSCSIDevnode
  -> RemoveDuplicateKMDF
  -> RemoveDuplicateSCSI
  -> Open YumeDiskKMDF
  -> QueryInfo target miniport
  -> SessionActive
```

## 2. 组件职责

## 2.1 TestApp

- 维护实际存储介质。
- 执行驱动包和设备实例自修复。
- 打开 `YumeDiskKMDF`，建立唯一控制会话。
- 周期性 heartbeat。
- 通过控制命令创建和删除磁盘。
- 为每个磁盘维护独立 read workers、write workers 和 `WRITE_ACK_BATCH` flush worker。
- 每个磁盘独立解释自己的 queue depth，不与其他磁盘共享数据面调度。
- 预投 read slots 和 write slots。
- 收到 read slot 后读取 medium 并发送 `READ_ACK`。
- 收到 write slot 后写入 medium 并发送 `WRITE_ACK_BATCH`。
- 负责 LBA overlap 下的数据一致性策略。

当前实现口径：

- 每个磁盘不是“单线程 slot engine”。
- 当前 App 数据面已经收敛为“每盘少量 read worker + 少量 write worker + 一个独立 write ACK flush worker”。
- worker 数量是有上限的小常数，并发度主要由 in-flight slot depth 提供，而不是靠 `Q` 个线程硬堆出来。

## 2.2 YumeDiskKMDF

### 对外职责

- 暴露 App 可枚举的自定义设备接口。
- 保证 active session 唯一。
- 感知 App file object 生命周期。
- 维护 heartbeat watchdog。
- 对 App IOCTL 做协议解析和 direct I/O buffer 获取。

### 对内职责

- 枚举并验证 `YumeDiskSCSI` storageport interface。
- 持有与 App file object 同生命周期的 miniport `HANDLE + FILE_OBJECT + DEVICE_OBJECT`。
- 代理短控制命令。
- 为 `POST_READ_SLOT` / `POST_WRITE_SLOT` 发起异步 slot transport，并在 miniport 完成后完成原始 App `WDFREQUEST`。
- session close 时按固定顺序清理：`RemoveAllDisks | SessionClose -> drain in-flight slot IO -> close miniport handle`。

`YumeDiskKMDF` 不拥有磁盘介质，不做数据面排队策略，也不保留旧 `WAIT_EVENT` inline payload 兼容层。

## 2.3 YumeDiskSCSI

### 对系统职责

- 响应标准 SCSI 指令。
- 以多 Target 方式暴露盘。
- 维护盘 presence 状态。
- 对只读盘向系统宣告写保护，并统一拒绝系统 WRITE 请求。
- 负责总线变化通知。
- 完成原始 READ/WRITE SRB。

### 对协议职责

- 接收 `YumeDiskKMDF` 转发的私有 miniport 命令。
- 每个 target 独立维护 read queue 和 write queue。
- 维护 `PendingReadsByEventId`。
- 维护 `PendingWritesByEventId` 和 write ACK bitmap。
- 消费 App 预投 slots。
- 处理 `READ_ACK` 和 `WRITE_ACK_BATCH`。
- session close、reset、remove、cancel 时取消 slot 并失败完成 pending SRB。

## 3. 设备发现

## 3.1 App 到 YumeDiskKMDF

App 通过 `SetupDiEnumDeviceInterfaces` 枚举 `YumeDiskKMDF` 自定义 GUID，不直接枚举 `GUID_DEVINTERFACE_STORAGEPORT`。

在进入接口枚举前，App 必须完成本地唯一性收敛：

- 通过命名互斥体或等效锁保证单进程。
- 通过 SetupAPI/ConfigMgr 枚举 `YumeDiskKMDF` Root/devnode 实例。
- 通过 SetupAPI/ConfigMgr 枚举 `YumeDiskSCSI` Root/devnode 实例。
- 缺失则安装驱动包并创建设备实例。
- 重复则删除多余实例。

## 3.1.1 驱动包安装

建议 App 侧使用标准驱动安装函数完成驱动包预装和安装。

推荐顺序：

1. 检查 INF 是否已在 Driver Store 中可用。
2. 若不可用，调用 `DiInstallDriver` 预装驱动包。
3. 后续若创建设备实例，PnP 绑定已在 Driver Store 中的匹配包。

相关官方文档：

- `DiInstallDriver`  
  https://learn.microsoft.com/en-us/windows/win32/api/newdev/nf-newdev-diinstalldriverw

## 3.1.2 根设备实例创建

当前两个驱动都是 Root 枚举模型，因此缺失设备实例时应由 App 显式创建 devnode。

建议机制：

1. `SetupDiCreateDeviceInfoList`
2. `SetupDiCreateDeviceInfo`
3. `SetupDiSetDeviceRegistryProperty` 写入硬件 ID
4. `SetupDiCallClassInstaller(DIF_REGISTERDEVICE)`

相关官方文档：

- `SetupDiCreateDeviceInfo`  
  https://learn.microsoft.com/en-us/windows/win32/api/setupapi/nf-setupapi-setupdicreatedeviceinfoa
- `DIF_REGISTERDEVICE`  
  https://learn.microsoft.com/en-us/previous-versions/windows/drivers/install/dif-registerdevice

## 3.1.3 重复设备清理

如果枚举结果多于一个，则按固定策略保留一个，删除其余实例。

建议删除接口：

- `SetupDiRemoveDevice`
- 或 `DiUninstallDevice`

本项目优先按“删除多余 devnode，不移除驱动包”处理。

相关官方文档：

- `SetupDiRemoveDevice`  
  https://learn.microsoft.com/en-us/windows/win32/api/setupapi/nf-setupapi-setupdiremovedevice

## 3.2 YumeDiskKMDF 到 YumeDiskSCSI

`YumeDiskKMDF` 在内核中枚举 `GUID_DEVINTERFACE_STORAGEPORT`，再通过以下方式过滤出目标 miniport：

- 设备实例 ID
- Service 名称
- FriendlyName
- `QueryScsiInfo` 私有命令验证签名和组件版本

最终以 `QueryScsiInfo` 返回固定签名和组件版本作为确认手段。

## 4. 磁盘模型

## 4.1 多盘方式

使用多 Target 模型：

- `Path = 0`
- `Target = 0..N-1`
- `Lun = 0`

不使用单 Target 多 LUN 模型。

## 4.2 Presence 模型

每个磁盘对象至少有以下状态：

- `Configured`
- `Present`
- `Removing`
- `Offline`

初始状态：

- `Configured = FALSE`
- `Present = FALSE`

额外约束：

- 整个系统只能存在一个有效 `YumeDiskSCSI` 适配器设备实例。
- 多块盘都挂在这一个适配器下，以多 Target 形式暴露。
- 无 App session 时所有盘都应为 not present。

## 4.3 容量与块大小

每块盘至少记录：

- `SectorSize`
- `SectorCount`
- `DiskSize`
- `TargetId`
- `Generation`

`READ_CAPACITY` 和 `READ_CAPACITY16` 必须都正确实现。超过 2TB 的介质必须保证 `READ_CAPACITY16` 返回真实容量，`READ_CAPACITY` 返回 `0xFFFFFFFF` 封顶语义。

## 5. 协议设计

协议分为两层：

- `TestApp <-> YumeDiskKMDF` 用户态控制协议。
- `YumeDiskKMDF <-> YumeDiskSCSI` 私有 miniport 协议。

用户态协议可以与 miniport 私有协议保持同构，但建议保留明确的结构边界。

## 5.1 命令分类

### 短控制命令

- `QUERY_KMDF_INFO`
- `QUERY_SCSI_INFO`
- `CREATE_DISK`
- `REMOVE_DISK`
- `REMOVE_ALL_DISKS`
- `QUERY_DEBUG_STATE`

### 数据面命令

- `POST_READ_SLOT`
- `POST_WRITE_SLOT`
- `READ_ACK`
- `WRITE_ACK_BATCH`
- `SUBMIT_SLOT`
- `CANCEL_SLOT`

旧 `WAIT_EVENT`、旧 `READ_REPLY`、旧 `WRITE_ACK` inline payload 语义不再作为目标实现保留。

## 5.2 slot 描述符

`YumeDiskKMDF` 传给 `YumeDiskSCSI` 的不应是大 payload，而是小 slot 描述符：

```c
typedef struct _YUMEDISK_SLOT_DESCRIPTOR {
    UINT64 SessionId;
    UINT64 SlotId;
    UINT32 SlotType;
    UINT32 TargetId;
    UINT64 KernelVa;
    UINT32 Capacity;
    UINT32 Flags;
} YUMEDISK_SLOT_DESCRIPTOR;
```

注意：

- `KernelVa` 只在对应 App request pending 期间有效。
- `YumeDiskSCSI` 完成 slot 后不得继续使用该 VA。
- `YumeDiskKMDF` 负责把 miniport completion 映射回原始 App `WDFREQUEST`。

## 5.3 READ slot 事件

```c
typedef struct _YUMEDISK_READ_SLOT_EVENT {
    UINT64 EventId;
    UINT32 TargetId;
    UINT32 Reserved0;
    UINT64 Lba;
    UINT32 BlockCount;
    UINT32 DataLength;
} YUMEDISK_READ_SLOT_EVENT;
```

规则：

- `EventId` 在 session 内单调递增，不复用。
- steady-state 下，`READ_ACK.EventId` 应命中 `PendingReadsByEventId`。
- 如果系统侧该 read 已经在 `SCSI` 内被取消、reset、remove 或 session close 清掉，`READ_ACK` 允许晚到；此时由 `SCSI` 判断为 stale / cancelled / not found，而不是要求 `App` 参与整条取消回传。
- `READ_ACK.DataLength` 必须匹配 pending read，除非 `IoStatus` 为失败。
- READ 失败只影响对应 READ SRB。

## 5.4 WRITE slot 事件

```c
typedef struct _YUMEDISK_WRITE_SLOT_HEADER {
    UINT64 EventId;
    UINT32 Seq;
    UINT32 TotalSeq;
    UINT32 TargetId;
    UINT32 Reserved0;
    UINT64 Lba;
    UINT32 ByteOffsetInWrite;
    UINT32 DataLength;
    UINT32 Flags;
    UINT32 Reserved1;
    UCHAR Data[1];
} YUMEDISK_WRITE_SLOT_HEADER;
```

规则：

- `EventId` 标识原始系统 WRITE。
- `Seq` 标识该 WRITE 内的 fragment 序号。
- `TotalSeq` 标识该 WRITE 总 fragment 数。
- `EventId + Seq` 是 write fragment 的唯一定位键。

## 5.5 WRITE_ACK_BATCH

写确认统一走独立 batch：

```c
typedef struct _YUMEDISK_WRITE_ACK_RANGE {
    UINT64 EventId;
    UINT32 SeqBase;
    UINT32 SeqCount;
    LONG IoStatus;
    UINT32 Reserved;
} YUMEDISK_WRITE_ACK_RANGE;

typedef struct _YUMEDISK_WRITE_ACK_BATCH {
    UINT32 RangeCount;
    UINT32 Reserved;
    YUMEDISK_WRITE_ACK_RANGE Ranges[1];
} YUMEDISK_WRITE_ACK_BATCH;
```

规则：

- `POST_WRITE_SLOT` 输入 payload 固定为空，不携带 ACK trailer。
- `WRITE_ACK_BATCH` 是唯一 ACK 入口。
- 每个 range 独立返回失败状态；单项失败不让整条控制命令失败。
- 如果对应系统 write 已经在 `SCSI` 侧被取消、remove 或 cleanup 清掉，late `WRITE_ACK_BATCH` 由 `SCSI` 逐项判定 stale / cancelled / not found；不要求 `App` 先收到一套全链路取消通知。
- ACK 必须校验 `EventId`、`SeqCount`、整数溢出、`TotalSeq`、`NextIssueSeq` 和重复 ACK。
- 只有 `AckedCount == TotalSeq` 时才能完成原始 WRITE SRB。

## 5.6 取消模型

取消边界统一收在 `YumeDiskSCSI`，不额外实现一套“系统取消一路追到 App，再由 App 回取消 ACK”的全链路复杂取消协议。

固定原则：

- 系统侧 `Cancel / Reset / Remove / SessionClose` 只要求追到 `SCSI` request completion 边界。
- `SCSI` 负责取消或失败完成原始系统 `READ/WRITE SRB`，并清理自己的 pending / issued request 状态。
- 已经发给 App 的 posted slot 不要求同步追回；App 可以继续按正常路径返回 `READ_ACK` 或 `WRITE_ACK_BATCH`。
- 当 App 的 ACK 晚到时，由 `SCSI` 再判断原始系统请求是否还存在。
- 如果原始请求已不存在，`SCSI` 只把这次 ACK 当作 stale / cancelled / not found 处理，不重新拉起系统取消链路，也不把整盘判死。

读路径语义：

- `READ_ACK` 到达时，如果 `EventId` 已不存在，`SCSI` 可以直接返回 `STATUS_NOT_FOUND` 或 `STATUS_CANCELLED`。
- 这类 late read ACK 只影响该事件本身，不影响后续读。

写路径语义：

- `WRITE_ACK_BATCH` 到达时，`SCSI` 对每个 range 独立判断是否仍然有效。
- stale / cancelled range 只在 batch result 中逐项返回失败，不让整条 batch 控制命令崩掉。
- 写错误或取消只影响对应原始 write，不直接把盘判死，也不阻断后续写。

## 6. 状态机

## 6.1 会话状态机

```text
NoSession
  -> AppOpen
SessionActive
  -> Running
Running
  -> HeartbeatLost / FileCleanup / AppClose
Closing
  -> RemoveAllDisks | SessionClose
  -> DrainInFlightSlotIo
  -> CloseMiniportHandle
ClosedLocked
```

进入 `ClosedLocked` 后，该 file object 的后续 IOCTL 直接失败，不允许用新的 heartbeat 或 slot 把 session 救活。

## 6.2 READ 状态机

```text
SystemReadSrb
  -> AllocateEventId
  -> PendingReadsByEventId
  -> CompletePostedReadSlotToApp
  -> READ_ACK
  -> CopyToSrbBuffer
  -> SrbCompleted
```

异常路径：

- `SessionClose`
- `DiskRemoved`
- `Cancel`
- `InvalidAck`

这些路径必须进入失败完成或明确取消。

## 6.3 WRITE 状态机

```text
SystemWriteSrb
  -> AllocateEventId
  -> SplitIntoSeqFragments
  -> PendingWritesByEventId
  -> CompletePostedWriteSlotsToApp
  -> WRITE_ACK_BATCH ranges
  -> BitmapAcked
  -> SrbCompleted when all seq acked
```

异常路径：

- `SessionClose`
- `DiskRemoved`
- `Cancel`
- `FragmentFailure`
- `InvalidAck`

这些路径不能留下 pending SRB 或已经失效的 slot 引用。

## 7. 并发与锁

锁设计统一采用“取走即解锁”原则：锁只保护共享状态和对象所有权转移，不包住实际工作。耗时动作必须在锁外执行。

固定流程：

```text
锁外准备
  -> 分配内存
  -> 校验不依赖共享状态的参数
  -> 计算长度、offset、seq、capacity

锁内转移状态
  -> 重新校验共享状态仍成立
  -> 从队列摘走 request / slot
  -> 修改计数、bitmap、state、generation
  -> 必要时把对象挂到本地 deferred list
  -> 立即解锁

锁外执行重活
  -> RtlCopyMemory
  -> IoCallDriver
  -> StorPortNotification(RequestComplete)
  -> WdfRequestComplete
  -> ControlFree / DiskFree
  -> KeWaitForSingleObject / WdfTimerStop(wait=TRUE)
  -> signal event
```

禁止项：

- 禁止在 `KSPIN_LOCK`、`WDFSPINLOCK` 内等待。
- 禁止在 `KSPIN_LOCK`、`WDFSPINLOCK` 内调用 `IoCallDriver`。
- 禁止在 `KSPIN_LOCK`、`WDFSPINLOCK` 内完成 SRB 或 WDFREQUEST。
- 禁止在 `KSPIN_LOCK`、`WDFSPINLOCK` 内做大块 `RtlCopyMemory`。
- 禁止在 `KSPIN_LOCK`、`WDFSPINLOCK` 内做可能触发复杂路径的分配/释放。
- 禁止持有 `WDFWAITLOCK` 后再等待 pending slot drain 或调用同步 miniport IOCTL。

允许项：

- 在锁内做链表插入/摘除。
- 在锁内做小字段赋值、计数增减、bitmap 标记。
- 在锁内读取当前 session id、state、generation。
- 在锁内把需要完成的对象移动到本地 list。

### 7.1 YumeDiskKMDF

至少维护以下同步对象：

- 全局 active session open 锁。
- file-bound session 生命周期锁。
- miniport handle / file object / device object 生命周期引用。
- in-flight async slot transport 引用。
- watchdog timer 状态。

当前锁类型边界：

- `OpenLock` 使用 `WDFSPINLOCK`，只保护 `OpenCount/OpenFileObject`。
- `SessionLock` 使用 `WDFWAITLOCK`，只用于 open、heartbeat、cleanup、watchdog、close miniport handle 等低频生命周期路径。
- `PendingSlotCount` 和 `InFlightRequestCount` 使用 interlocked 计数，cleanup/watchdog 用 `KEVENT` drain。

`SessionLock` 不得成为 `POST_READ_SLOT` / `POST_WRITE_SLOT` 的高频准入锁。数据面准入应采用原子 state/generation + pending ref：

```text
read active state/generation
InterlockedIncrement(PendingSlotCount)
read active state/generation again
if changed:
    InterlockedDecrement(PendingSlotCount)
    fail request
```

cleanup/watchdog 顺序：

```text
lock SessionLock
state = Closing
unlock

wake transport runtime
send RemoveAllDisks | SessionClose
wait PendingSlotCount == 0

lock SessionLock
close miniport handle
state = Closed/Locked
unlock
```

不能持有 `SessionLock` 等待 `PendingSlotZeroEvent`，否则 slot completion 和 cleanup 容易形成等待环。

后续 `KMDF async slot transport` 对象池也必须遵守同一规则：

- submit queue/free list 可以用 spin lock 保护链表。
- enqueue/dequeue 时锁内只移动对象。
- worker 从共享队列批量取一小段到本地 list 后立即解锁。
- `IoCallDriver`、IRP completion 处理、`WdfRequestComplete` 和对象释放都在锁外。
- 对象池满时初版直接返回 busy 给 App 重投，不在 KMDF 内无限等待池对象。
- 该优化按“激进更新原则”重建 KMDF transport，不在旧 per-slot IRP/work item 路径上继续叠补丁。
- 新 runtime 落地后，旧 per-slot transport 路径必须删除，不保留新旧双轨或兼容开关。

### 7.2 YumeDiskSCSI

至少维护以下同步对象：

- adapter/session 状态锁：`KSPIN_LOCK`。
- 每个 target 自己的 `ReadQueueLock`：`KSPIN_LOCK`。
- 每个 target 自己的 `WriteQueueLock`：`KSPIN_LOCK`。
- 每个 disk buffer 指针保护锁：`KSPIN_LOCK`。
- `NextEventId` 的 interlocked counter 或专用锁。

SCSI 处在 Storport/miniport 路径上，per-target queue 继续使用 `KSPIN_LOCK`，不改成 wait lock。优化方向是缩短锁内工作、减少锁内线性查找，而不是扩大临界区或换成等待锁。

READ dispatch：

```text
lock ReadQueueLock
find pending read
pop posted read slot
mark SlotIssued / counters
unlock

write read event into App buffer
complete slot SRB
free slot object
```

WRITE dispatch：

```text
lock WriteQueueLock
find writable request
pop posted write slot
seq = NextIssueSeq++
unlock

copy SRB fragment to App write slot
complete slot SRB
free slot object
```

READ_ACK：

```text
lock ReadQueueLock
find request by EventId
remove request from pending list
update counters
unlock

validate data length / KernelVa
copy App buffer to original SRB
complete original READ SRB
free request
```

WRITE_ACK_BATCH：

```text
for each range:
    lock WriteQueueLock
    find write request
    check seq bounds / duplicate bitmap
    set ack bits
    if request complete or failed:
        remove from pending
        move request to local deferred completion list
    unlock

write batch result outside lock
complete WRITE_ACK_BATCH ioctl SRB outside lock
complete deferred original WRITE SRB outside lock
```

remove / cleanup：

```text
lock target read/write locks
move all pending requests and posted slots to local lists
reset queue counters/state
unlock

complete all local SRBs
complete all local slot SRBs
free all local objects
notify bus change
```

disk storage buffer：

```text
lock BufferLock
oldBuffer = Disk->Buffer
Disk->Buffer = NULL
unlock

DiskFree(oldBuffer)
```

长期原则：

- 队列摘取和状态变更在锁内。
- 大数据 copy 在锁外。
- SRB completion 在锁外。
- App request completion 在锁外。
- 请求推进采用 `drain while available`，不要只推进一个请求就等待下一次唤醒。
- 多盘不能共享 adapter 级全局数据面锁或共享队列。
- 如果锁内线性查找成为热点，优先增加 ready list / pending-unissued list / eventId index，不扩大锁范围。

## 8. 热插拔与清理策略

删除盘时动作顺序必须固定：

1. 标记目标盘 `Present = FALSE`。
2. 从 per-target pending 表中摘除相关 READ/WRITE 请求。
3. 取消或完成相关 posted slots。
4. 失败完成相关 pending SRB。
5. 通知 Storport 总线变化。

该动作等效于“设备突然拔除”，不要求文件系统优雅卸载。

session close 时必须覆盖全部 target，并额外关闭 KMDF 持有的 miniport handle。

## 9. 错误处理

### 会话丢失

- 删除全部盘。
- 取消全部 pending slot。
- 失败完成全部 pending SRB。
- 关闭 miniport handle。
- 锁定 session。

### 启动收敛失败

- 不进入工作态。
- 不允许建盘。
- 记录清晰错误并退出。

### READ_ACK 错误

- 只失败完成对应 READ SRB。
- 不把磁盘判死。
- 不影响后续独立 READ。
- 如果系统侧该 read 已经不存在，允许返回 `STATUS_NOT_FOUND` 或 `STATUS_CANCELLED`。

### WRITE_ACK_BATCH 错误

- 单个 range 错误记录到 batch result。
- stale/cancelled ACK 可返回 `STATUS_NOT_FOUND` 或 `STATUS_CANCELLED`。
- 不因为单个 ACK range 失败让整条控制命令失败。

## 10.1 当前性能结论

当前已经确认过的性能结论：

- 单盘从早期约 `20 MB/s` 的低吞吐恢复到数百 MB/s，热路径 print 删除后可显著回升。
- “每盘单线程 slot engine”会明显压低单盘吞吐，即使多盘总吞吐还能继续扩展。
- 当前 App 数据面改成“每盘少量 read/write workers + 独立 write ACK flush worker”后，单盘顺序吞吐可恢复到约 `600 MB/s` 级别。
- 在此基础上如果仍然出现“吞吐够高，但单盘内核 CPU 很高”，优先怀疑 `KMDF async slot transport` 的每-slot 固定开销，而不是再回头怀疑 App worker 模型。
- `Q32T2` 压测过程中，如果宿主机执行截图等外部扰动操作，虚拟机内吞吐可能短时从约 `500 MB/s` 掉到约 `300 MB/s`；当前观察该现象可自行恢复，因此暂按环境调度扰动处理，不定性为当前缺陷。

### 目标盘不存在

- 直接返回设备不存在类错误。

### 协议版本不匹配

- 当前不再单独维护 `ProtocolVersion`。
- 统一改为“单一组件版本即协议门槛”：
  - `YumeDiskKMDF`
  - `YumeDiskSCSI`
  - `AppKernel`
  都只维护一个 `VersionBe`。
- `VersionBe` 的格式固定为 `A.B.C.D` 四段十进制，每段范围 `0-255`，底层存储为大端序 `UINT32`。
- 版本号同时代表：
  - 二进制版本
  - 协议版本
  - 行为契约版本
- 只要版本不一致，就默认协议不兼容，不再额外引入第二套 `ProtocolVersion` 判定。
- 兼容性只通过 `VersionBe` 相等判定，不再额外引入 `Features`、能力位或其他并行准入规则。
- `KMDF` 和 `SCSI` 的信息查询应拆成独立来源，不允许再把“当前返回的是谁的版本”混在一个模糊 `QueryInfo` 里：
  - `QueryKmdfInfo`
  - `QueryScsiInfo`
- `AppKernel` 打开会话时必须执行严格版本校验：
  1. 打开 `YumeDiskKMDF` 控制设备。
  2. 查询 `KMDF.VersionBe`。
  3. 查询 `SCSI.VersionBe`。
  4. 校验 `KMDF.VersionBe == AppKernel.VersionBe`。
  5. 校验 `SCSI.VersionBe == AppKernel.VersionBe`。
  6. 全部通过才允许进入工作态。
- 任何一种情况都必须直接失败，不允许降级继续运行：
  - `KMDF` 版本查询失败
  - `SCSI` 版本查询失败
  - `KMDF` 版本不等于 `AppKernel`
  - `SCSI` 版本不等于 `AppKernel`
- 版本号应只有一个共享真源头，例如共享头中的单一常量；`KMDF / SCSI / AppKernel` 必须都从同一处编译进来，不允许三处手工维护。
- 对外发行后，只要协议字段、命令语义、错误语义或兼容性边界发生变化，就必须升级该版本；不允许“代码已变，但版本不变”。
- 当前仍处于未发行阶段，因此版本继续固定为 `0.1.0.0`，直到首次正式发行前不单独滚动版本号。

### devtidy 版本规则

- `devtidy` 不参与 `KMDF / SCSI / AppKernel` 的运行时版本相等校验。
- 但 `devtidy` 自己也不再拆分“工具版本”和“JSONL schema 版本”两套编号。
- `devtidy` 只维护单一版本，例如 `v17`。
- 这个版本同时代表：
  - `devtidy` 可执行工具版本
  - `devtidy` JSONL 输出协议版本
- 将来只要 JSONL 字段、事件名或语义发生不兼容变化，就直接升级 `devtidy` 版本，不再另外维护独立 `schema=v1/v2` 体系。

## 10. 验证口径

最终多盘口径下，核心验证不再是旧 `WAIT_EVENT` 是否闭环，而是：

- `POST_READ_SLOT` / `READ_ACK` 可以完成系统 probe read 和 steady-state read。
- `POST_WRITE_SLOT` / `WRITE_ACK_BATCH` 可以完成顺序和并发 write。
- `Q1T1 / Q8 / Q32` 不出现 100% 无读写假死。
- Ctrl+C、`RemoveAllDisks`、App 退出和 heartbeat 超时都能清理 pending slot 与 pending SRB。
- 多盘并发时不出现 adapter 级全局队列串扰。
- 热路径日志关闭后再采集吞吐和 CPU 数据。
