# 项目概述

## 1. 项目目标

YumeDisk 要实现一个由用户态宿主主导的数据盘型虚拟磁盘系统。Windows 看到的是 `YumeDiskSCSI` 暴露的 Storport SCSI target；真实存储介质、读写一致性策略和磁盘生命周期由用户态宿主负责。

驱动与用户态数据面内核的长期收口边界由三层组成：

- `AppKernel`：用户态数据面内核，负责会话、slot、ACK、事件和 per-disk worker 运行时。
- `YumeDiskKMDF`：控制驱动，负责自定义设备接口、单会话、watchdog、file cleanup 和 App direct I/O buffer 到 miniport 的 transport。
- `YumeDiskSCSI`：Storport miniport，负责 SCSI 枚举、系统 READ/WRITE SRB、per-target 队列和原始 SRB completion。

当前仓内正式宿主主线是：

- `windows/tauri-client`
- `windows/BackendRust`
- `AppKernel`
- `YumeDiskKMDF`
- `YumeDiskSCSI`

并行辅助宿主/工具线包括：

- `windows/cpp-cli` + `windows/BackendCore`
- `windows/rust-cli`

早期设计文档里的 `MyApp`、`MyCtl`、`MyDisk` 是历史逻辑名；已经不存在的 `windows/TestApp/` 不再作为当前事实引用。

最终目标不是传统内核自包含 ramdisk，而是 App-owned media 架构：

```text
Windows storage stack
  -> YumeDiskSCSI per-target READ/WRITE queue
  -> YumeDiskKMDF session/direct-I/O slot transport
  -> AppKernel per-disk runtime
  -> host-owned medium and config
```

## 2. 使用范围

该系统只面向以下场景：

- 普通数据盘。
- 应用软件直接读写的数据卷。
- 测试型、专用型、受控环境中的虚拟存储。

明确不支持以下场景：

- 系统盘。
- 启动盘。
- 分页盘。
- 依赖用户态回包才能维持系统稳定的核心卷。

## 3. 设计原则

### 3.1 App 不在线即无盘

只要没有有效用户态会话，`YumeDiskSCSI` 就不应暴露任何数据盘。宿主正常退出、异常退出、heartbeat 超时或 KMDF file cleanup 都必须收敛到同一结果：清盘、取消 pending slot、失败完成 pending SRB。

### 3.2 启动即自检与自修复

驱动侧长期目标要求宿主在启动前具备环境收敛能力：

- 检查 `YumeDiskKMDF` 和 `YumeDiskSCSI` 驱动包。
- 检查两个 Root/devnode 设备实例是否存在且唯一。
- 缺失时安装驱动包并创建设备实例。
- 重复时删除多余实例。
- 最后只打开 `YumeDiskKMDF` 自定义设备接口。

补充说明：

- 这是驱动侧和宿主侧的长期目标边界。
- 当前正式桌面主线已经具备“只打开 `YumeDiskKMDF` 控制接口”和版本校验能力。
- 当前正式桌面主线尚未把“驱动包安装、devnode 创建/去重”作为已落地事实写死。

### 3.3 控制入口和 SCSI 入口分离

用户态宿主不直接枚举或打开 `GUID_DEVINTERFACE_STORAGEPORT`。`YumeDiskKMDF` 对外提供稳定控制入口，并在内核中通过 `QueryScsiInfo` 等私有命令确认目标 `YumeDiskSCSI` miniport；兼容性本身只通过组件版本相等判定。

### 3.4 数据面只有一条链路

最终数据面只允许 app-owned slot queue：

- `POST_READ_SLOT`
- `POST_WRITE_SLOT`
- `READ_ACK`
- `WRITE_ACK_BATCH`

旧 `WAIT_EVENT -> inline data -> READ_REPLY/WRITE_ACK` 模型只作为历史基线和故障排查背景，不再作为目标实现。

### 3.5 状态归属唯一

- `YumeDiskKMDF` 只拥有 session、watchdog、miniport handle 和 App request 生命周期。
- `YumeDiskSCSI` 最终只拥有 per-target posted slots、pending reads/writes、eventId 和 SRB completion。
- 用户态宿主最终只拥有介质、配置和盘运行时真状态；`AppKernel` 负责 per-disk workers、queue depth、READ_ACK 和 WRITE_ACK_BATCH flush。

不要回到 adapter 级共享数据面队列，也不要让多盘共享 App 全局 worker pool。

### 3.6 驱动不做 LBA overlap 一致性

读写在驱动层完全并发处理。驱动不做 LBA overlap 检测、不做读写排序；重叠区间的一致性由 App 自己负责。

## 4. 运行流程

### 4.1 启动阶段

1. 宿主准备会话配置和盘运行时状态。
2. `AppKernel` 枚举并打开 `YumeDiskKMDF` 自定义设备接口。
3. `YumeDiskKMDF` 创建唯一 active session。
4. `YumeDiskKMDF` 打开同会话生命周期的 `YumeDiskSCSI` miniport handle，并通过 `QueryScsiInfo` 确认目标。
5. `AppKernel` 开始 heartbeat。

当前正式桌面主线中，这条链路具体表现为：

1. `tauri-client` 恢复 `.yumedisk/client.json`
2. `BackendRust` 恢复 `DiskRuntimeStore`
3. `BackendRust` 调用 `AppKernel::AkOpen`
4. `AppKernel` 通过 `YumeDiskKMDF` 建立会话

### 4.2 创建磁盘

1. 宿主请求创建目标 `TargetId`。
2. `AppKernel` 为该盘创建 runtime 并启动独立 worker 组。
3. `YumeDiskKMDF` 将短控制命令代理给 `YumeDiskSCSI`。
4. `YumeDiskSCSI` 标记 target present 并通知总线变化。
5. `AppKernel` 持续预投 read/write slots。

### 4.3 正常读路径

1. Windows 向 `YumeDiskSCSI` 发起 READ SRB。
2. `YumeDiskSCSI` 分配 `eventId`，把请求挂入该 target 的 `PendingReadsByEventId`。
3. `YumeDiskSCSI` 消费一个 posted read slot，填写 `eventId/lba/count`，经 `YumeDiskKMDF` 完成给 App。
4. `AppKernel` 调用宿主介质读视图读取数据。
5. `AppKernel` 通过 `READ_ACK` 把数据回送给驱动。
6. `YumeDiskSCSI` 将数据复制到原始 SRB buffer 并完成 READ SRB。

### 4.4 正常写路径

1. Windows 向 `YumeDiskSCSI` 发起 WRITE SRB。
2. `YumeDiskSCSI` 分配 `eventId`，按 write slot 容量拆成 `seq/totalSeq` fragment。
3. `YumeDiskSCSI` 消费 posted write slots，把 fragment 数据填入 App slot。
4. `AppKernel` 调用宿主暂存写入当前 fragment。
5. `AppKernel` 用独立 `WRITE_ACK_BATCH` 按 `eventId + seq range` 落账。
6. `YumeDiskSCSI` 用 bitmap 记录 ACK；全部 fragment 落账后完成原始 WRITE SRB。

### 4.5 退出与清理

1. 宿主正常关闭、异常退出、heartbeat 超时或 KMDF file cleanup 触发 session close。
2. `YumeDiskKMDF` 向 `YumeDiskSCSI` 发送 `RemoveAllDisks | SessionClose`。
3. `YumeDiskSCSI` 清盘，取消 pending slots，失败完成 pending SRB。
4. `YumeDiskKMDF` drain in-flight slot transport，关闭 miniport handle，并锁定该 file object 后续 IOCTL。

## 5. 交付结果

项目完成后应具备以下结果：

- 宿主可稳定打开会话并动态创建和删除多个虚拟盘。
- 系统可将这些虚拟盘识别为普通数据盘。
- 读写路径由 App-owned slot queue 主导。
- 宿主退出后虚拟盘立即消失，pending I/O 被明确失败完成或取消。
- 多盘按多 Target 方式暴露，不使用多 LUN 方案。
- `Q1T1 / Q8 / Q32` 压测不再出现 100% 无读写假死、盘被误判死亡、会话残留或多盘全局串扰。
